> - **มี mini-phase แทรกคั่น 3 จุด** (เฟสเล็ก ทำเหมือนเฟสจริงทุกอย่าง สอนทักษะคู่ขนานที่ไม่ใช่ฟีเจอร์ใหม่): **Mini-Phase A — Debugging & DevTools** (หลัง Phase 2), **Mini-Phase B — Git แบบทีม: branch + Pull Request** (หลัง Phase 3), **Mini-Phase C — Async JavaScript + ดึง API ภายนอก** (หลัง Phase 6). สอนตามลำดับที่แทรกไว้ อย่าข้าม.

> - เวลาที่ประเมินเป็น "ชั่วโมงคร่าวๆ" สำหรับมือใหม่ ปรับตามจังหวะผู้เรียนได้ ไม่ใช่เส้นตาย.
>

## Phase 0 — Setup & ปฐมนิเทศ

### 🎯 เป้าหมายของเฟส
ให้ผู้เรียน "เห็นเว็บของตัวเองรันในเบราว์เซอร์" ภายในวันแรก พร้อมเข้าใจภาพรวมว่าเว็บทำงานยังไง และมีเครื่องมือพร้อมใช้ทั้งหมด. เป้าหมายคือลดความกลัว terminal และสร้างความมั่นใจว่า "ฉันทำได้".

### 🧱 สิ่งที่จะสร้างเพิ่มใน TaskFlow
- โปรเจกต์ Next.js เปล่าชื่อ `taskflow` ที่รันได้ในเครื่อง
- repository บน GitHub ที่ push โค้ดขึ้นไปแล้ว 1 commit

### 📚 คอนเซปต์ที่จะสอน (พร้อมคำอธิบายสั้นๆ)
- **เว็บทำงานยังไง (ภาพรวม):** เบราว์เซอร์ (client) ส่งคำขอ → เซิร์ฟเวอร์ (server) ตอบกลับ. อธิบายด้วยภาพ "ลูกค้าสั่งอาหาร ↔ ครัวทำส่ง".
- **Frontend vs Backend:** frontend = สิ่งที่ผู้ใช้เห็นและกดในเบราว์เซอร์; backend = ตรรกะ+ข้อมูลที่ทำงานบนเซิร์ฟเวอร์. Next.js ทำได้ทั้งสองอย่างในโปรเจกต์เดียว (จุดเด่นสำคัญ).
- **HTTP เบื้องต้น:** request/response, URL คืออะไร, ทำไมต้องมี `localhost:3000`.
- **Node.js คืออะไร:** ตัวรัน JavaScript นอกเบราว์เซอร์ ทำให้เขียน backend ด้วยภาษาเดียวกับ frontend ได้.
- **Next.js / React คืออะไร (สั้นๆ พอเห็นภาพ):** React = วิธีสร้าง UI เป็นชิ้นๆ (component); Next.js = framework ที่ห่อ React ให้ทำเว็บจริงได้ครบ (routing, server, build).
- **Terminal / command line:** หน้าต่างพิมพ์คำสั่งสั่งคอมพิวเตอร์ — สอนแค่ที่จำเป็น (`cd`, `ls`/`dir`, รันคำสั่ง, หยุดด้วย Ctrl+C).
- **Git & GitHub:** Git = ระบบบันทึกประวัติเวอร์ชันโค้ด (เหมือน save game); GitHub = ที่เก็บโค้ดออนไลน์ + เป็นหน้าโชว์ผลงาน.
- **VS Code:** โปรแกรมเขียนโค้ด สอน layout เบื้องต้น (file explorer, editor, integrated terminal).

### 👣 ลำดับขั้นจับมือทำ
1. **ปฐมนิเทศ 10 นาที:** วาด/อธิบายภาพ client–server และบอกว่าวันนี้จะได้เห็นเว็บตัวเองรัน. ถามผู้เรียนว่าเคยใช้ terminal ไหม เพื่อปรับระดับ.
2. **ติดตั้งเครื่องมือ** ตาม `SETUP.md`: Node.js (LTS), VS Code, Git. ตรวจว่าติดตั้งสำเร็จด้วยคำสั่งเช็คเวอร์ชัน (คำสั่งจริงดู `SETUP.md`). อธิบายว่าตัวเลขที่ขึ้นมาแปลว่า "ติดตั้งแล้ว".
3. **ทัวร์ VS Code 5 นาที:** เปิดโฟลเดอร์ว่าง, ชี้ส่วนต่างๆ, เปิด integrated terminal (จะใช้ตัวนี้ตลอดหลักสูตร).
4. **สร้างโปรเจกต์ Next.js แรก:** รันคำสั่ง create Next.js app (คำสั่ง+ตัวเลือกตอบ prompt ดู `SETUP.md`). อธิบายว่าแต่ละไฟล์/โฟลเดอร์ที่เกิดขึ้นคืออะไรแบบคร่าวๆ (ยังไม่ต้องลึก).
5. **รัน dev server:** สั่งรัน แล้วเปิด `http://localhost:3000` ในเบราว์เซอร์ — **โมเมนต์สำคัญ: เห็นหน้าแรกของตัวเอง**. ให้ผู้เรียนลองแก้ข้อความบนหน้าแล้ว save ดูว่ามันเปลี่ยนทันที (แนะนำคำว่า hot reload).
6. **รู้จัก terminal จากของจริง:** อธิบาย Ctrl+C เพื่อหยุด server, รันใหม่อีกครั้ง.
7. **git init + commit แรก:** อธิบาย commit = "เซฟจุดที่พอใจ". ทำ `git init`, ดูสถานะ, add, commit.
8. **สร้าง repo บน GitHub + push:** สร้าง repo ชื่อ `taskflow` (ดูขั้นตอน auth/คำสั่งใน `SETUP.md`), เชื่อม remote, push. เปิดหน้า GitHub ให้ผู้เรียนเห็นโค้ดตัวเองออนไลน์.

### ✅ Checkpoint/แบบฝึกหัด
- เปิด `localhost:3000` เองได้โดยไม่ต้องให้ช่วย และอธิบายได้ว่า "ทำไมต้องรัน dev server ก่อนถึงจะเปิดหน้าเว็บได้".
- แก้ข้อความบนหน้าแรกเป็นชื่อตัวเอง เช่น "สวัสดี ฉันชื่อ ___" แล้ว save เห็นเปลี่ยนในเบราว์เซอร์.
- ตอบสั้นๆ ได้ว่า frontend กับ backend ต่างกันยังไง 1 ประโยค.
- เปิดหน้า GitHub repo ของตัวเองได้.

## Phase 1 — React & UI พื้นฐาน

### 🎯 เป้าหมายของเฟส
ให้ผู้เรียนเข้าใจหัวใจของ React คือ "component" และสร้างหน้าตา (UI) ของ task list แบบ static (ยังกดใช้งานไม่ได้ แต่หน้าตาเหมือนของจริง) ด้วย Tailwind CSS. เน้นให้เห็นว่า UI ประกอบจากชิ้นเล็กๆ ที่นำกลับมาใช้ซ้ำได้.

### 🧱 สิ่งที่จะสร้างเพิ่มใน TaskFlow
- หน้า TaskFlow ที่มี header, ช่องกรอกงาน (ยังไม่ทำงาน), และรายการ task หลายอันแบบ static
- แตกหน้าเป็น component: `Header`, `TaskList`, `TaskItem`, `AddTaskForm`
- จัดสไตล์ด้วย Tailwind ให้ดูสะอาดตา

### 📚 คอนเซปต์ที่จะสอน (พร้อมคำอธิบายสั้นๆ)
- **JSX:** วิธีเขียน "HTML ปนใน JavaScript" — หน้าตาเหมือน HTML แต่เป็นโค้ด. สอนข้อต่างสำคัญ (`className` แทน `class`, ปิด tag เสมอ, ใส่ค่าด้วย `{}`).
- **Component:** ฟังก์ชันที่ return UI 1 ชิ้น เอามาใช้ซ้ำได้ เหมือน "บล็อกตัวต่อ". ชื่อขึ้นต้นตัวใหญ่เสมอ.
- **Props:** วิธีส่งข้อมูลจาก component แม่ไปลูก เหมือน "ส่งพารามิเตอร์ให้ฟังก์ชัน" ทำให้ component เดียวแสดงข้อมูลต่างกันได้.
- **การ compose component:** ประกอบ component ย่อยขึ้นเป็นหน้าใหญ่.
- **render list ด้วย `.map()`:** เปลี่ยน array ของข้อมูลให้กลายเป็น UI หลายชิ้น (ปูทางไปแสดง task หลายอัน).
- **Tailwind CSS:** วิธีจัดสไตล์ด้วย utility class เช่น `flex`, `p-4`, `text-lg`, `rounded` — เขียนสไตล์ตรงใน JSX ไม่ต้องสลับไฟล์ CSS.

### 👣 ลำดับขั้นจับมือทำ
1. **ทบทวน Phase 0 สั้นๆ** แล้วเคลียร์หน้าแรกให้เหลือโครงเปล่าๆ เพื่อเริ่มสร้าง TaskFlow.
2. **JSX ก้าวแรก:** ให้แสดงหัวข้อ "TaskFlow" บนหน้า อธิบาย JSX จากสิ่งที่เพิ่งพิมพ์.
3. **สร้าง component แรก `Header`:** แยกหัวข้อออกเป็นไฟล์ component แล้ว import มาใช้ — อธิบายว่าทำไมแยก (อ่านง่าย, ใช้ซ้ำ).
4. **สร้าง `TaskItem` 1 ชิ้น:** กล่องแสดงงาน 1 งาน (checkbox ปลอม + ข้อความ + ปุ่มลบปลอม). ยังไม่ต้องทำงาน เน้นหน้าตา.
5. **ส่งข้อมูลด้วย props:** ทำให้ `TaskItem` รับ prop `title` แล้วแสดงข้อความตามที่ส่งเข้าไป — วางหลายอันด้วยข้อความต่างกัน เพื่อให้เห็นพลังของ props.
6. **สร้าง `TaskList` + render ด้วย `.map()`:** เก็บ task เป็น array (hardcode ไว้ก่อน) แล้ว map ออกมาเป็น `TaskItem` หลายชิ้น. (แนะนำ `key` เบาๆ ว่าจะอธิบายเต็มใน Phase 2.)
7. **สร้าง `AddTaskForm` (static):** ช่อง input + ปุ่ม "เพิ่ม" หน้าตาเหมือนจริง แต่ยังไม่ทำงาน.
8. **จัดสไตล์ด้วย Tailwind:** ค่อยๆ ใส่ class ทีละส่วน (spacing, สี, มุมโค้ง, hover) ให้เห็นหน้าตาดีขึ้นเรื่อยๆ. เน้นว่าแก้ทีละนิดแล้วดูผลทันที.

> 🎨 **หมายเหตุสำหรับผู้สอน — ใช้ skill `frontend-design` ช่วยตอนออกแบบ UI:** ระหว่างขั้นที่ 4–8 (ออกแบบ `TaskItem`/`AddTaskForm` และจัดสไตล์ Tailwind) ให้เรียกใช้ skill **`frontend-design`** ช่วยคิด layout/สเปซ/สี/ลำดับสายตา เพื่อให้หน้า TaskFlow ดูสะอาดเป็นมืออาชีพตั้งแต่แรก. ใช้เป็น "ผู้ช่วยออกแบบ" — ผู้เรียนยังเป็นคนพิมพ์ JSX/class เองทุกบรรทัด และผู้สอนอธิบาย "ทำไมถึงจัดแบบนี้" ควบคู่ไป ไม่ใช่ปล่อยให้ skill generate รวดเดียวจนผู้เรียนตามไม่ทัน. (skill นี้ใช้อีกครั้งตอน Phase 6 ตอนขัดเงา UI.)

### ✅ Checkpoint/แบบฝึกหัด

## Phase 2 — State & Interactivity

### 🎯 เป้าหมายของเฟส
ทำให้ TaskFlow "ใช้งานได้จริง" ในฝั่ง frontend — เพิ่ม/แก้ไขชื่องาน/ติ๊กเสร็จ/ลบ task ได้ และข้อมูลไม่หายเมื่อรีเฟรช (เก็บใน localStorage). หัวใจคือเข้าใจ **state** ว่าเป็น "ความจำของ component" ที่เปลี่ยนแล้วหน้าจออัปเดตตาม.

### 🧱 สิ่งที่จะสร้างเพิ่มใน TaskFlow
- เพิ่ม task จากฟอร์มได้จริง
- แก้ไขชื่อ task ที่มีอยู่ได้ (inline edit — กดแก้แล้วชื่อเปลี่ยน)
- ติ๊ก checkbox เพื่อ mark done/undone ได้
- ลบ task ได้
- ข้อมูล task ถูกเก็บใน localStorage (รีเฟรชแล้วยังอยู่)

### 📚 คอนเซปต์ที่จะสอน (พร้อมคำอธิบายสั้นๆ)
- **State (`useState`):** ข้อมูลที่ component "จำ" ไว้ พอเปลี่ยน React จะ re-render หน้าให้อัตโนมัติ. ต่างจากตัวแปรธรรมดาตรงที่เปลี่ยนแล้วหน้าจอเปลี่ยนตาม.
- **Event handling:** ตอบสนองการกระทำผู้ใช้ เช่น `onClick`, `onChange`, `onSubmit`.
- **Controlled form (controlled input):** ผูกค่าใน input กับ state ทำให้ React เป็น "เจ้าของ" ค่าในฟอร์ม.
- **อัปเดต state แบบ immutable:** สร้าง array/object ชุดใหม่แทนการแก้ของเดิม (ใช้ spread `...`, `.map()`, `.filter()`) — อธิบายว่าทำไม React ต้องการแบบนี้.
- **List & keys:** ทำไมแต่ละ item ใน list ต้องมี `key` ที่ไม่ซ้ำ (ช่วย React track ว่าอันไหนเปลี่ยน).
- **`useEffect`:** รันโค้ด "ข้างเคียง" หลัง render เช่น sync กับ localStorage; แนะนำแนวคิด dependency array แบบเบาๆ.
- **localStorage:** ที่เก็บข้อมูลเล็กๆ ในเบราว์เซอร์ (อยู่ที่ฝั่ง client). อธิบายว่าเป็นของชั่วคราว เดี๋ยว Phase 4 จะย้ายไป database จริง.
- **Client Component (`"use client"`):** บอก Next.js ว่า component นี้ต้องทำงานในเบราว์เซอร์เพราะใช้ state/event. (ปูทางไป Phase 3 server vs client.)

> 📦 **กล่องเสริม: "hydration" คืออะไร (อ่านก่อนเจอคำเตือนด้านล่าง)**
> ก่อนหน้านี้เราคิดว่าหน้าเว็บถูกสร้างในเบราว์เซอร์ที่เดียว แต่จริงๆ แล้ว Next.js ทำงาน 2 จังหวะ: **(1)** ครั้งแรก Next.js สร้างหน้า (HTML) ขึ้นที่ **ฝั่ง server** ก่อน เพื่อให้หน้าโผล่ไว **(2)** แล้วเบราว์เซอร์ (**client**) ค่อยมา "รับช่วงต่อ" ทำให้หน้ากดใช้งานได้ — ขั้นรับช่วงต่อนี้เรียกว่า **hydration** (เหมือนเติมน้ำให้โครงแห้งๆ กลายเป็นของใช้งานได้).
> ปัญหา **hydration mismatch / hydration error** เกิดเมื่อ "หน้าที่ server สร้าง" กับ "หน้าที่ client สร้างตอนรับช่วงต่อ" **ออกมาไม่ตรงกัน** React จะงงและฟ้อง error. ตัวอย่างคลาสสิกคือการอ่าน `localStorage` ตอนตั้งค่าเริ่มต้น state เพราะ **ฝั่ง server ไม่มี `localStorage`** (มันอยู่แค่ในเบราว์เซอร์) ค่าสองฝั่งจึงต่างกันแน่นอน.
> ตอนนี้เข้าใจแค่ "server สร้างก่อน client รับช่วงต่อ ถ้าสองฝั่งได้ค่าไม่ตรงกันจะ error" ก็พอ — เราจะเรียน **server vs client เต็มๆ ใน Phase 3** ตอนนั้นภาพจะชัดขึ้นมาก. ส่วนวิธีหลบ error นี้อยู่ในลำดับขั้นข้อ 7 (ขั้นเก็บลง localStorage) ด้านล่าง.

### 👣 ลำดับขั้นจับมือทำ
1. **แปลง array hardcode เป็น state:** เอา task list เข้าไปอยู่ใน `useState` อธิบายว่าตอนนี้ข้อมูล "มีชีวิต" แล้ว. ใส่ `"use client"` พร้อมอธิบายสั้นๆ ว่าทำไมต้องมี.
2. **เพิ่ม task ได้จริง:** ทำ controlled input ใน `AddTaskForm`, จับ `onSubmit`, เพิ่ม task ใหม่เข้า state ด้วย spread. ทดสอบเพิ่มงานเห็นโผล่ทันที.
3. **toggle done:** ทำ `onClick`/`onChange` ที่ checkbox ให้สลับสถานะ done ของ task นั้นด้วย `.map()`. ใส่ style ขีดฆ่าเมื่อ done.
4. **ลบ task:** ปุ่มลบเรียก handler ที่ใช้ `.filter()` เอา task นั้นออก.
5. **แก้ไขชื่อ task (inline edit):** เพิ่มปุ่ม "แก้ไข" ใน `TaskItem`. กดแล้วสลับจากข้อความธรรมดาเป็นช่อง input (controlled input ผูกกับ state เช่น `editingId` + `editText`) เพื่อพิมพ์ชื่อใหม่ → กด "บันทึก" เรียก handler ที่ใช้ `.map()` อัปเดต `title` ของ task ที่ id ตรงกัน (สร้าง object ใหม่แบบ immutable) แล้วกลับมาแสดงเป็นข้อความเหมือนเดิม. มีปุ่ม "ยกเลิก" เพื่อออกจากโหมดแก้ไขโดยไม่บันทึก. ย้ำแนวคิดเดิม: เราไม่ได้แก้ของเก่าตรงๆ แต่สร้าง array ชุดใหม่ที่มี task ตัวที่แก้แล้ว.
6. **ใส่ key ให้ถูกต้อง:** ทบทวน list & keys ให้แต่ละ task มี id ที่ไม่ซ้ำ (เช่น สร้าง id ตอนเพิ่ม) แล้วใช้เป็น key. อธิบายอาการ bug ถ้าใช้ index เป็น key.
7. **เก็บลง localStorage ด้วย `useEffect`:** เมื่อ task เปลี่ยน → เซฟลง localStorage; ตอนโหลดหน้า → อ่านกลับมาใส่ state. ทดสอบรีเฟรชแล้วข้อมูลอยู่.
   > ⚠️ **ป้องกัน hydration mismatch (สำคัญสำหรับ Next.js — ทบทวน "กล่องเสริม: hydration คืออะไร" ด้านบนก่อน):** อย่าอ่าน localStorage ตอนกำหนดค่าเริ่มต้นของ `useState` (เช่น `useState(JSON.parse(localStorage...))`) เพราะตอน Next.js สร้างหน้าที่ฝั่ง server ยังไม่มี localStorage (มันมีแค่ในเบราว์เซอร์) ค่าที่ server สร้างกับที่ client สร้างจึงไม่ตรงกัน เกิด error. ใช้ pattern นี้แทน: เริ่ม state เป็น **array ว่าง** (เพื่อให้ server กับ client ได้ค่าเริ่มต้นตรงกัน) → `useEffect` **ตัวแรก** (รันครั้งเดียว, dependency `[]` คือรันหลังเบราว์เซอร์รับช่วงต่อแล้ว) โหลดข้อมูลจาก localStorage มาใส่ state → `useEffect` **ตัวที่สอง** (dependency `[tasks]`) บันทึกลง localStorage ทุกครั้งที่ state เปลี่ยน. ถ้าผู้เรียนเจอ error คำว่า "hydration" ให้กลับมาเช็คจุดนี้ก่อน. (ยังไม่ต้องเข้าใจ server/client ลึกตอนนี้ — เดี๋ยวเต็มใน Phase 3.)
8. **เก็บกวาด edge case เบาๆ:** กันเพิ่ม task ว่าง, trim ช่องว่าง, และกันบันทึกชื่อที่แก้แล้วเป็นค่าว่าง — ฝึกคิดเรื่อง input ของผู้ใช้.

### ✅ Checkpoint/แบบฝึกหัด

- อธิบายได้ว่าทำไมต้องสร้าง array ใหม่แทนการ `push` เข้า array เดิม (และตอนแก้ไขก็สร้าง object ใหม่ ไม่แก้ของเดิมตรงๆ).
- แบบฝึกหัด: เพิ่มตัวนับ "เหลืออีก ___ งานที่ยังไม่เสร็จ" โดยคำนวณจาก state (ฝึกอ่านค่า state มาคำนวณ).
- อธิบายความต่างของ "ตัวแปรธรรมดา" กับ "state" ได้.

### 📚 คอนเซปต์ที่จะสอน (อธิบายสั้น ๆ จากศูนย์)

- **DevTools (Developer Tools) คืออะไร:** เครื่องมือสำหรับนักพัฒนาที่ "ติดมากับเบราว์เซอร์อยู่แล้ว" (Chrome, Edge, Firefox มีหมด) — ไม่ต้องติดตั้งอะไรเพิ่ม. มันคือหน้าต่างที่เปิดแล้วเห็น "เบื้องหลัง" ของหน้าเว็บ: โครงสร้างหน้า, ข้อความ error, ค่าตัวแปร, การโหลดข้อมูล. เปรียบเหมือน "เปิดฝากระโปรงรถดูเครื่องยนต์" — ปกติผู้ใช้ทั่วไปเห็นแค่หน้าเว็บสวยๆ แต่ dev เปิดฝาดูข้างในได้.
- **แท็บ Console:** หน้าต่างที่เบราว์เซอร์ใช้ "พูดคุย" กับเรา — มันพ่นข้อความ error (ตัวหนังสือสีแดง) และ warning (สีเหลือง) ออกมาที่นี่. และเป็นที่ที่ `console.log(...)` ของเราจะไปโผล่. คิดง่ายๆ ว่า Console = "ช่องแชตระหว่างโค้ดของเรากับเบราว์เซอร์".
- **`console.log()`:** คำสั่งที่ใส่ในโค้ดเพื่อ "สั่งพิมพ์ค่าออกมาดู". เช่น `console.log(tasks)` แปลว่า "เฮ้ ช่วยพิมพ์ค่าของ `tasks` ตอนนี้ให้ดูหน่อย" — ค่าจะไปโผล่ในแท็บ Console. นี่คือเครื่องมือ debug ที่ dev ทุกระดับใช้ตลอด (ไม่ใช่แค่มือใหม่) เพราะมันบอกเราได้ว่า "ตอนนี้ในตัวแปรมีอะไรอยู่จริงๆ" ซึ่งบ่อยครั้งไม่ตรงกับที่เราคิด.
- **แท็บ Elements:** แสดง **DOM** ของหน้าเว็บ. *DOM (Document Object Model) คืออะไร:* คือ "ต้นไม้โครงสร้างของหน้าเว็บ" ที่เบราว์เซอร์สร้างขึ้นจากโค้ดเรา — กล่องซ้อนในกล่อง (`<div>` ข้างใน `<div>` ฯลฯ). JSX ที่เราเขียนใน React สุดท้ายกลายเป็น DOM นี้แหละ. ในแท็บนี้เราจะเห็นว่า `TaskItem` ของเรากลายเป็น HTML หน้าตายังไง และ Tailwind class (เช่น `flex p-4 rounded`) กลายเป็น style จริงๆ อะไรบ้าง.
- **แท็บ Network (เกริ่นพอ):** แสดง "การส่งข้อมูลเข้า-ออก" ระหว่างเบราว์เซอร์กับเซิร์ฟเวอร์ — ไฟล์ไหนโหลด, ใช้เวลานานแค่ไหน, สำเร็จหรือ error. **ตอนนี้ TaskFlow ยังไม่มี backend/database (เก็บใน localStorage ล้วน) จึงยังไม่ค่อยมีอะไรน่าสนใจในแท็บนี้** — เราแค่เปิดให้รู้จักไว้ก่อน. มันจะกลายเป็นเครื่องมือสำคัญตอน Phase 4 (Database) เมื่อเริ่มมีการรับ-ส่งข้อมูลกับเซิร์ฟเวอร์จริง.
- **React DevTools (extension เสริม):** DevTools ปกติเห็นแค่ DOM (HTML สุดท้าย) แต่ **ไม่เห็นโลกของ React** (component, state, props). React DevTools คือส่วนเสริมที่ติดตั้งเพิ่มในเบราว์เซอร์ แล้วจะมีแท็บใหม่ให้ส่อง "ต้นไม้ของ component" ได้ — กดที่ `TaskItem` แล้วเห็นเลยว่ามัน prop อะไรเข้ามา, ตัวที่มี state เก็บ `tasks` ไว้ในนั้นมีค่าอะไร. เหมาะมากเวลาเดาไม่ออกว่า "ข้อมูลที่ส่งให้ component ถูกไหม".
- **วิธีอ่าน error เป็นขั้นเป็นตอน (หัวใจของเฟส):** error ส่วนใหญ่ "บอกวิธีแก้ในตัวมันเอง" ถ้าเราอ่านเป็น. 4 ขั้น: **(1)** อ่านข้อความบรรทัดบนสุด (คือสรุปว่า "ผิดอะไร"), **(2)** หาชื่อไฟล์ + เลขบรรทัด (คือ "ผิดที่ไหน"), **(3)** ไล่ stack trace คร่าวๆ (คือ "โค้ดวิ่งมาทางไหนก่อนพัง"), **(4)** ถ้ายังไม่เข้าใจ ค่อยใช้ `console.log` หรือ breakpoint ส่องค่าตรงจุดนั้น.
- **Breakpoint (แบบง่ายสุด):** จุดที่เราสั่งให้โค้ด "หยุดแช่ไว้ตรงนี้" เพื่อเราจะได้ส่องดูว่าตอนนั้นค่าตัวแปรเป็นอะไร เหมือนกดปุ่ม pause หนังเพื่อดูเฟรมนั้นชัดๆ. เฟสนี้สอนแค่ระดับเริ่มต้น (เห็นภาพว่ามันคืออะไร + ลองหยุด 1 ที) — ไม่ต้องลึก.
- **mindset: error เป็นเรื่องปกติ:** dev อาชีพเจอ error ทั้งวันทุกวัน. error ไม่ได้แปลว่าเราโง่หรือทำพัง — มันคือเบราว์เซอร์ "บอกเราตรงๆ ว่าอะไรไม่ตรง" ซึ่งดีกว่าพังเงียบๆ มาก. ทักษะที่แยก dev มือใหม่กับมือเก๋าไม่ใช่ "ไม่เจอ error" แต่คือ "อ่าน error แล้วแก้ได้เร็ว".

### 👣 ลำดับขั้นจับมือทำ

1. **เปิดใจก่อนเปิดเครื่องมือ (2–3 นาที):** ก่อนแตะอะไร คุยกับผู้เรียนสั้นๆ ว่า "ตอนทำ Phase 2 เธอเจอข้อความสีแดงใช่ไหม — นั่นปกติมาก ฉัน (และ dev ทุกคนบนโลก) เจอแบบนั้นทุกวัน. วันนี้เราจะติดเครื่องมือไว้อ่านมันให้ออก." ตั้งเป้าให้ผู้เรียนรู้สึกโล่ง ไม่ใช่กดดัน.

2. **เปิด DevTools ครั้งแรก:** ให้ TaskFlow รันอยู่ที่ `localhost:3000` (ถ้า server ปิดอยู่ให้สั่งรันก่อน — ทบทวนจาก Phase 0). จากนั้นเปิด DevTools ด้วยวิธีใดก็ได้:
   - กดปุ่ม **F12** บนคีย์บอร์ด, **หรือ**
   - **คลิกขวา** ที่ไหนก็ได้บนหน้าเว็บ → เลือก **Inspect** (บางเบราว์เซอร์ภาษาไทยเขียนว่า "ตรวจสอบ").
   จะมีหน้าต่างโผล่ขึ้นมา (มักอยู่ด้านขวาหรือด้านล่างของหน้าจอ). ให้ผู้เรียนสังเกตว่า "ข้างบนมีแถวแท็บ: Elements, Console, Network, ..." — บอกว่าเราจะไล่ดูทีละแท็บ ไม่ต้องตกใจที่มันเยอะ. *(ทำไม: ให้ผู้เรียนเปิดเองได้ก่อน เพราะต้องเปิดบ่อยมากตลอดหลักสูตร.)*

3. **รู้จักแท็บ Console + อ่าน error ที่มีอยู่:** กดที่แท็บ **Console**. ดูว่ามีข้อความสีแดง (error) หรือสีเหลือง (warning) ค้างอยู่ไหม (จาก Phase 2 อาจยังมีอยู่). ถ้ามี ให้หยุดอ่านด้วยกันตาม "4 ขั้นอ่าน error" (สอนเต็มในข้อ 8). ถ้าไม่มี error ก็ดี — บอกว่า "Console ว่างๆ แบบนี้แปลว่าตอนนี้ไม่มีปัญหาที่เบราว์เซอร์อยากบอก". *(ทำไม: ให้ผู้เรียนเห็นว่า Console คือช่องที่เบราว์เซอร์ใช้สื่อสาร.)*

4. **ลองใช้ `console.log` ส่อง state จริงของ TaskFlow:** เปิดไฟล์ component ที่เก็บ state `tasks` (component หลักของ TaskFlow). เพิ่มบรรทัดส่องค่าก่อนบรรทัด `return`:
   ```jsx
   console.log("ค่า tasks ตอนนี้:", tasks);
   ```
   save → กลับไปดูแท็บ Console → จะเห็นข้อความ + array ของ task โผล่ออกมา. ให้ผู้เรียน **กดลูกศรเล็กๆ หน้า array เพื่อกางดูข้างใน** — เห็นแต่ละ task มี `id`, `title`, `done` อะไรบ้าง. ลองเพิ่ม task ใหม่บนหน้าเว็บ แล้วดูว่า Console พิมพ์ค่าใหม่ออกมา (เพราะ component re-render). *(ทำไม: นี่คือทักษะ debug ที่ใช้บ่อยสุด — "ส่องดูว่าในตัวแปรมีอะไรจริงๆ" บ่อยครั้ง bug มาจากค่าไม่ตรงกับที่เราคิด.)* **อย่าลืมเตือน:** debug เสร็จแล้วค่อยลบ `console.log` ออกก่อน commit (ไม่ใช่เรื่องผิด แต่ทำให้โค้ดสะอาด).

5. **รู้จักแท็บ Elements + ส่อง Tailwind:** กดแท็บ **Elements**. จะเห็นโครงสร้าง HTML (DOM) ของหน้า เป็นกล่องซ้อนกล่อง. ให้ลองกดไอคอน "ลูกศรชี้" (มุมซ้ายบนของ DevTools) แล้ว **คลิกที่ task อันหนึ่งบนหน้าเว็บ** — DevTools จะกระโดดไปไฮไลต์ HTML ของ `TaskItem` อันนั้นให้. ชี้ให้เห็นว่า:
   - JSX ที่เราเขียนกลายมาเป็น HTML จริงหน้าตาแบบนี้
   - class Tailwind (เช่น `flex`, `p-4`, `rounded`) ปรากฏใน attribute `class` และทางขวามีช่อง **Styles** บอกว่าแต่ละ class กลายเป็น CSS อะไร (เช่น `p-4` → `padding: 1rem`).
   ลองให้ผู้เรียน **ติ๊ก/เลิกติ๊ก style ในช่อง Styles** เพื่อเห็นหน้าเปลี่ยนสดๆ (การแก้ตรงนี้เป็นแค่ทดลองชั่วคราว รีเฟรชแล้วหาย — ไม่กระทบโค้ดจริง). *(ทำไม: ช่วยให้ผู้เรียนเข้าใจว่า "Tailwind class → CSS จริง" และใช้แท็บนี้เดาปัญหา layout/สไตล์ได้.)*

6. **เปิดแท็บ Network ดูคร่าวๆ (เกริ่นพอ):** กดแท็บ **Network** แล้วรีเฟรชหน้า. จะเห็นรายการไฟล์ที่เบราว์เซอร์โหลด (HTML, JS, CSS, รูป). บอกผู้เรียนสั้นๆ ว่า "นี่คือบันทึกการรับ-ส่งข้อมูลกับเซิร์ฟเวอร์ — ตอนนี้ TaskFlow ของเรายังเก็บข้อมูลใน localStorage ในเครื่องล้วนๆ จึงยังไม่มีการส่งข้อมูล task เข้า-ออกให้ดู. **เราจะกลับมาใช้แท็บนี้จริงจังตอน Phase 4** ที่เริ่มมี database." ไม่ต้องลงรายละเอียด — แค่ให้รู้ว่ามีแท็บนี้และไว้ทำอะไร. *(ทำไม: ปูทาง ไม่ให้ผู้เรียนตกใจตอนเจอใน Phase 4 แต่ไม่ overload ตอนนี้.)*

7. **ติดตั้ง React DevTools + ส่อง state/props:**
   1. **ติดตั้ง extension** "React Developer Tools" ลงในเบราว์เซอร์ (ดูลิงก์/ขั้นตอนล่าสุดใน `SETUP.md` — เป็น extension จาก Chrome Web Store / Firefox Add-ons. ผู้สอนยืนยันชื่อ store ตามเบราว์เซอร์ที่ผู้เรียนใช้). หลังติดตั้งต้องรีเฟรชหน้า TaskFlow 1 ครั้ง.
   2. เปิด DevTools (F12) จะเห็น **แท็บใหม่ชื่อ "Components"** (และ "Profiler") โผล่เพิ่มมาในแถวแท็บ. กดที่ **Components**.
   3. จะเห็น "ต้นไม้ของ React component" — เลื่อนหา/คลิกที่ `TaskItem` อันหนึ่ง → ทางขวาจะโชว์ **props** ที่ส่งเข้ามา (เช่น `title`, `done`). คลิกที่ component ที่ถือ state `tasks` → จะเห็นค่า state ตรงนั้น.
   4. ลองเพิ่ม/ติ๊ก task บนหน้าเว็บ แล้วดูค่าใน Components อัปเดตตาม.
   *(ทำไม: DevTools ปกติเห็นแค่ DOM แต่ React DevTools เห็น "โลกของ React" — เวลา bug เกิดจาก "ข้อมูลที่ส่งให้ component ผิด" เครื่องมือนี้ช่วยได้ตรงจุดที่สุด.)*

8. **สอน "4 ขั้นอ่าน error" บนตัวอย่างจริง (สอนวิธีคิด):** ใช้กรอบนี้ทุกครั้งที่เจอ error สีแดง — ทั้งใน Console และในหน้าจอ overlay error ของ Next.js:
   1. **อ่านบรรทัดบนสุด** = "ผิดอะไร" เช่น `Cannot read properties of undefined (reading 'title')` แปลคร่าวๆ ว่า "พยายามอ่าน `.title` จากของที่เป็น `undefined` (ไม่มีค่า)".
   2. **หาชื่อไฟล์ + เลขบรรทัด** = "ผิดที่ไหน" เช่น `TaskItem.jsx:12` → ไปเปิดไฟล์นั้น บรรทัดนั้น. ใน DevTools ชื่อไฟล์มักเป็นลิงก์กดไปได้เลย.
   3. **ไล่ stack trace คร่าวๆ** = "โค้ดวิ่งมาทางไหนก่อนพัง" — รายการบรรทัดที่ไล่ลงมาคือลำดับฟังก์ชันที่เรียกต่อๆ กัน. **มือใหม่ดูแค่ 1–2 บรรทัดบนสุดที่เป็นไฟล์ของเราพอ** (บรรทัดที่ชี้ไปไฟล์ของ library ข้ามได้).
   4. **ส่องค่าตรงจุดนั้น** = ถ้ายังไม่เข้าใจ ใส่ `console.log` ก่อนบรรทัดที่พัง เพื่อดูว่าค่าที่คาดว่าจะมีจริงๆ มันเป็นอะไร.
   ย้ำกับผู้เรียน: **อย่ากลัวข้อความยาวๆ** — เราอ่านแค่บรรทัดบนสุดกับชื่อไฟล์ก่อน ที่เหลือค่อยๆ ดู.

9. **แบบฝึกหัดหัวใจของเฟส — "จงใจทำ bug แล้วใช้ DevTools จับ":** ให้ผู้เรียนทำ bug ขึ้นมาเอง 1 ตัว (เลือก 1 แบบ) แล้วไล่จับด้วย 4 ขั้นข้างบน:
   - **แบบ A (พิมพ์ชื่อ prop/ตัวแปรผิด):** ใน `TaskItem` เปลี่ยน `task.title` เป็น `task.titel` (สะกดผิด) → หน้าจะแสดงผลเพี้ยนหรือขึ้น error. ให้ผู้เรียนเปิด Console/หน้า error อ่านตาม 4 ขั้นจนเข้าใจว่า "เพราะ field ชื่อ `titel` ไม่มีจริง" → แก้กลับเป็น `task.title`.
   - **แบบ B (เรียกของที่ยังไม่มี):** ลองอ่าน `tasks[0].title` ทั้งที่ตอนเริ่มต้น `tasks` เป็น array ว่าง → เจอ `Cannot read properties of undefined` → ไล่หาว่าผิดที่บรรทัดไหน → เข้าใจว่า "ยังไม่มี task เลย จึงไม่มี `tasks[0]`" → แก้กลับ.
   หลังจับเจอและแก้กลับสำเร็จ ให้ผู้เรียนพูดออกมาเป็นคำของตัวเองว่า "error บอกอะไร และฉันรู้ได้ยังไงว่าต้องไปแก้ตรงไหน". **นี่คือ Checkpoint ที่สำคัญที่สุด** — ถ้าผู้เรียนทำได้เอง แปลว่าเฟสนี้บรรลุเป้า.

10. **(ทางเลือก) ลองตั้ง breakpoint 1 ที:** ในแท็บ **Sources** (Chrome/Edge) เปิดไฟล์ component ของเรา → คลิกที่ "เลขบรรทัด" ด้านซ้ายของบรรทัดที่อยากหยุด → จะมีจุดสีน้ำเงินขึ้น (นั่นคือ breakpoint). ทำ action บนหน้าเว็บที่จะวิ่งผ่านบรรทัดนั้น (เช่นกดปุ่มเพิ่ม task) → โค้ดจะ "หยุดแช่" ตรงนั้น แล้วเอาเมาส์ไปชี้ที่ตัวแปรเพื่อดูค่าได้. กดปุ่ม ▶ (resume) เพื่อให้โค้ดวิ่งต่อ. บอกผู้เรียนว่า "ตอนนี้แค่รู้ว่ามันมีอยู่และทำอะไรพอ — `console.log` ก็เพียงพอสำหรับงานส่วนใหญ่แล้ว." *(ทำไม: ให้เห็นว่ามีเครื่องมือที่ทรงพลังกว่ารออยู่ แต่ไม่กดดันให้ต้องใช้ตอนนี้.)*

11. **เก็บกวาด:** ลบ `console.log` ที่ใส่ไว้ทดลองออกให้หมด, แก้ bug ที่จงใจทำให้กลับเป็นปกติ, ตรวจว่า TaskFlow ยังทำงานครบเหมือนจบ Phase 2 (เพิ่ม/แก้/ติ๊ก/ลบ + รีเฟรชแล้วข้อมูลอยู่). *(ทำไม: เฟสนี้ไม่ควรทิ้งร่องรอยหรือทำให้แอปพัง — เราแค่มาฝึกเครื่องมือ.)*

### ✅ Checkpoint/แบบฝึกหัด
- เปิด DevTools เองได้ 2 วิธี (F12 และ คลิกขวา > Inspect) และชี้ได้ว่าแท็บ Console / Elements / Network ใช้ดูอะไร.
- ใส่ `console.log(tasks)` แล้วเปิด Console กางดูข้างใน array ได้ว่ามี task อะไรบ้าง (พิสูจน์ว่าใช้ `console.log` ส่องค่าเป็น).
- ในแท็บ Elements คลิก task บนหน้าเว็บแล้วบอกได้ว่า Tailwind class `p-4` (หรือ class อื่น) กลายเป็น CSS อะไร.
- เปิดแท็บ Components (React DevTools) แล้วบอก props/state ของ component หนึ่งได้.
- **(สำคัญที่สุด) จงใจทำ bug 1 ตัว → อ่าน error ตาม 4 ขั้น (ข้อความบนสุด → ชื่อไฟล์+บรรทัด → stack trace → ส่องค่า) → หาเจอว่าผิดตรงไหน → แก้กลับได้ด้วยตัวเอง.**
- ตอบเป็นคำพูดได้ว่า "ทำไม error ถึงเป็นเรื่องปกติของ dev" และอธิบายได้ว่าเวลาเจอข้อความแดงๆ จะเริ่มอ่านจากตรงไหนก่อน.

เฟสนี้ปกติไม่เปลี่ยนโค้ดแอป (debug เสร็จแล้วเก็บกวาดกลับ) จึงอาจ "ไม่มีอะไรให้ commit" ก็ได้ — ถ้าเป็นแบบนั้นไม่ต้องฝืน commit เปล่าๆ. แต่ถ้าผู้เรียนสร้างไฟล์โน้ตช่วยจำ (`DEBUGGING-NOTES.md`) ให้ commit ด้วยข้อความ:


## Phase 3 — Next.js App Router

### 🎯 เป้าหมายของเฟส
เปลี่ยนจาก "หน้าเดียว" เป็น "เว็บหลายหน้า" จริงๆ ด้วย App Router ของ Next.js และเข้าใจความต่างที่สำคัญที่สุดของ Next.js: **Server Component vs Client Component**. จัดโครงโปรเจกต์ให้สะอาดพร้อมโตต่อ.

### 🧱 สิ่งที่จะสร้างเพิ่มใน TaskFlow
- หลายหน้า: หน้าหลัก (tasks), หน้า About, โครงหน้า Dashboard (เปล่าๆ รอ Phase 6)
- `layout` ร่วม (header/nav อยู่ทุกหน้า) + navigation ด้วย `Link`
- จัดโครงโฟลเดอร์ component ให้เป็นระเบียบ
- ใส่ metadata/title พื้นฐานให้แต่ละหน้า

### 📚 คอนเซปต์ที่จะสอน (พร้อมคำอธิบายสั้นๆ)
- **App Router & file-based routing:** โครงสร้างโฟลเดอร์ใน `app/` กลายเป็น URL อัตโนมัติ — ไฟล์ `page.js` = หน้าหนึ่งหน้า.
- **`layout.js`:** โครงที่ครอบทุกหน้า (เช่น header/footer) ไม่ต้องเขียนซ้ำ.
- **`Link` component:** เปลี่ยนหน้าแบบ client-side navigation (เร็ว ไม่โหลดใหม่ทั้งหน้า) ต่างจาก `<a>` ธรรมดา.
- **Server Components vs Client Components:** ค่าเริ่มต้นใน App Router คือ Server Component (รันบนเซิร์ฟเวอร์ เบา ปลอดภัย เข้าถึง data ได้ตรง); ต้องใส่ `"use client"` เฉพาะเมื่อใช้ state/event/hook ในเบราว์เซอร์. นี่คือคอนเซปต์หัวใจของ Next.js สมัยใหม่.
- **Metadata/SEO เบื้องต้น:** ตั้ง `title`/`description` ของหน้า เพื่อ search engine และ tab เบราว์เซอร์.
- **โครงโปรเจกต์:** แยกโฟลเดอร์ `components/` ออกจาก `app/`, ตั้งชื่อไฟล์ให้สื่อความหมาย.

### 👣 ลำดับขั้นจับมือทำ
1. **ทำความเข้าใจ `app/` ที่มีอยู่:** เปิด `layout.js` กับ `page.js` ที่ Next.js สร้างให้ อธิบายว่ามันคือ layout ราก + หน้าแรก.
2. **สร้างหน้า About:** สร้างโฟลเดอร์ `about/page.js` แล้วเปิด `/about` ในเบราว์เซอร์ — เห็นว่า "สร้างโฟลเดอร์ = สร้างหน้า".

8. **จัดระเบียบไฟล์:** ย้าย component ไปโฟลเดอร์ `components/` ปรับ import ให้เรียบร้อย อธิบายว่าโครงดีช่วยให้โตต่อง่าย.

## Mini-Phase B — Git แบบทีม: branch + Pull Request (แทรกหลัง Phase 3)

> **mini-phase นี้คืออะไร:** mini-phase สั้น ๆ ที่แทรกคั่นระหว่าง Phase 3 กับ Phase 4 เพื่อยกระดับการใช้ Git ของผู้เรียนจาก "เซฟตรง ๆ คนเดียว" ไปเป็น "ทำงานแบบทีมมืออาชีพ" ด้วย branch และ Pull Request. ฟอร์แมตเหมือนเฟสอื่นใน `CURRICULUM.md` ทุกอย่าง.
>
> **สำหรับผู้สอน (Claude Code ปลายทาง) อ่านก่อนเริ่ม:**
> - ณ จุดนี้ผู้เรียนทำ `git init` / `git add` / `git commit` / `git push` แบบ "เส้นตรง" (ทุก commit ต่อกันบน `main` เส้นเดียว) เป็นแล้วตั้งแต่ Phase 0–3 และมีโค้ด TaskFlow หลายไฟล์ขึ้นไปบน GitHub แล้ว. mini-phase นี้ **ไม่เพิ่มฟีเจอร์ให้ TaskFlow** แต่เปลี่ยน *วิธีทำงานกับโค้ด*.
> - **ห้ามเทรวดเดียว** — พาทำทีละ sub-step ตาม "👣 ลำดับขั้นจับมือทำ" ให้ผู้เรียนพิมพ์คำสั่งเองทุกครั้งและดูผลลัพธ์บนหน้าจอ/บน GitHub ก่อนไปต่อ.
> - **อธิบาย "ทำอะไร" คู่ "ทำไม" เสมอ** โดยเฉพาะ "ทำไมทีมถึงไม่ commit ลง main ตรง ๆ" — นี่คือหัวใจของ mini-phase นี้.
> - mini-phase นี้ทำได้ทั้ง **วิธีเว็บ GitHub** และ **`gh` CLI** (ติดตั้งไว้แล้วตั้งแต่ `SETUP.md`). พาทำ **วิธีเว็บก่อน** เพราะเห็นภาพชัดสุด แล้วค่อยโชว์ทางลัด `gh` ทีหลัง.
> - **ห้าม pin เลขเวอร์ชัน** — ถ้าต้องเช็คคำสั่ง `gh` ล่าสุดให้อ้างอิง `SETUP.md`.

### 🎯 เป้าหมายของเฟส
ให้ผู้เรียนเลิกกลัวคำว่า "branch" และ "Pull Request" แล้วใช้มันเป็นนิสัย. จบ mini-phase นี้ผู้เรียนจะ:
- เข้าใจว่า **branch คือ "จักรวาลคู่ขนานของโค้ด"** ที่ลองของใหม่ได้โดยไม่พังของเดิมบน `main`
- สร้าง/สลับ branch, commit บน branch, push branch ขึ้น GitHub ได้เอง
- เปิด **Pull Request (PR)**, review โค้ดของตัวเอง, merge เข้า `main`, แล้วกลับมา `main` แล้ว `pull` ให้เครื่องตัวเอง sync
- **แก้ merge conflict ได้** (เพราะเราจะจงใจสร้าง conflict ให้เจอในที่ปลอดภัย แล้วพาแก้ทีละขั้น)
- เข้าใจ `.gitignore` ลึกขึ้น ว่าทำไม **ห้าม** commit `node_modules/`, `.env`, `dev.db`
- รับ "กติกาทำงานมืออาชีพ" ไปใช้ต่อ: **1 ฟีเจอร์ = 1 branch = 1 PR** กับ Phase 4–8 ที่เหลือ

> **ทำไมแทรกตรงนี้:** Phase 4 เป็นต้นไปจะเริ่มเป็นงานก้อนใหญ่ (database, auth, ฟีเจอร์เยอะ). ถ้าฝึก branch + PR ให้ชินตั้งแต่ตอนนี้ ผู้เรียนจะทำ Phase ที่เหลือแบบมีระเบียบ มี "ปุ่ม undo ระดับฟีเจอร์" และมีประวัติงานที่เล่าตอนสัมภาษณ์ได้ว่า "ทำงานเป็นทีมเป็น".

### 🧱 สิ่งที่จะทำ/สร้างใน TaskFlow
mini-phase นี้เน้น *กระบวนการ* ไม่ใช่ฟีเจอร์ — แต่จะมีของจริงเกิดขึ้นบน repo:
- branch ใหม่อย่างน้อย 2 อันบน GitHub (เช่น `docs/notes` กับ branch ทดลองทำ conflict)
- Pull Request ที่ผู้เรียน **เปิดเอง, review เอง, merge เอง** อย่างน้อย 1 อัน (มีร่องรอยใน tab "Pull requests" ของ repo)
- ไฟล์เล็ก ๆ ที่ปลอดภัยชื่อ `NOTES.md` ที่เราใช้เป็น "สนามซ้อม" สร้างและแก้ conflict — **ไม่แตะโค้ดแอปจริง** เพื่อไม่ให้ TaskFlow พังระหว่างฝึก. *(จงใจใช้ `NOTES.md` ไม่ใช่ `README.md` เพราะ `README.md` ตัวจริงจะเขียนอย่างเป็นทางการตอน Phase 8 — ถ้าซ้อม conflict บน README ตอนนี้จะเหลือ README ขยะค้างไปจนถึงตอนนั้น.)*
- `.gitignore` ที่เราเปิดดู ทำความเข้าใจ และยืนยันว่าครอบคลุม `node_modules/`, `.env*`, `*.db` (เตรียมพร้อมก่อนเข้า Phase 4 ที่จะมี `dev.db` และ `.env` จริง)

### 📚 คอนเซปต์ที่จะสอน (อธิบายสั้น ๆ จากศูนย์)

- **ทบทวน commit คืออะไร (จากศูนย์):** ที่ผ่านมา commit = "ถ่ายรูปเซฟสถานะโค้ดทั้งโปรเจกต์ ณ ตอนนั้น" เหมือนเซฟเกม. ที่ผ่านมาเราเซฟต่อ ๆ กันเป็น **เส้นตรงเส้นเดียว** บนเส้นชื่อ `main`.

- **branch คืออะไร — "จักรวาลคู่ขนานของโค้ด":** ลองนึกว่าโค้ดของเราคือเส้นเวลา (timeline) ของหนัง. `main` คือเส้นเวลาหลัก "ของจริง" ที่ต้องใช้ได้เสมอ. **branch คือการแตกเส้นเวลาออกไปอีกเส้นหนึ่ง** เพื่อ "ลองของใหม่" — เช่นลองเพิ่มฟีเจอร์ ลองแก้สี ลองรื้อโครง — โดยที่เส้น `main` เดิม **ไม่เปลี่ยนแปลงเลย**. ถ้าลองแล้วดี เราค่อย "รวม" (merge) เส้นทดลองกลับเข้า `main`. ถ้าลองแล้วพัง ก็แค่ทิ้งเส้นนั้น `main` ไม่เคยรู้เรื่องด้วยซ้ำ. **นี่คือ "ปุ่มลองผิดลองถูกที่ปลอดภัย" ของโปรแกรมเมอร์.**

- **ทำไมทีมไม่ commit ลง `main` ตรง ๆ:** ในงานจริงหลายคนแก้โค้ดก้อนเดียวกันพร้อมกัน. ถ้าทุกคนเขียนทับ `main` พร้อมกันจะชนกันมั่ว และถ้าใครเผลอ push ของพัง คนทั้งทีมพังตาม. กติกาคือ "อยากเปลี่ยน `main` ต้องผ่าน **branch + Pull Request** ให้คนอื่น (หรือตัวเองในอนาคต) ได้ดูก่อน". `main` จึง "สะอาดและใช้ได้เสมอ".

- **HEAD / branch ที่ active อยู่:** Git จำว่าตอนนี้เรา "ยืนอยู่บน" branch ไหน (เรียกว่า `HEAD`). commit ใหม่จะถูกบันทึกลง branch ที่เรายืนอยู่ตอนนั้น — เพราะงี้ "ยืนผิด branch แล้ว commit" คือพลาดที่เจอบ่อย เราจะสอนเช็คก่อนเสมอ.

- **Pull Request (PR) คืออะไร:** PR คือ **คำขออย่างเป็นทางการว่า "ขอเอาโค้ดจาก branch ของฉัน รวมเข้า `main` นะ"** ผ่านหน้าเว็บ GitHub. มันไม่ใช่คำสั่ง Git โดยตรง แต่เป็นฟีเจอร์ของ GitHub ที่เปิด "หน้าพูดคุย" ให้เห็นว่า branch นี้เปลี่ยนอะไรไปบ้าง (diff), ให้คนคอมเมนต์/อนุมัติ, แล้วกดปุ่มรวมได้. **ทีมใช้ PR เพราะมันคือจุดที่เกิด code review** — โอกาสจับ bug และเรียนรู้กันก่อนของจะเข้า `main`.

- **review PR ของตัวเอง:** แม้ทำคนเดียว การ "อ่าน diff ของตัวเองก่อน merge" ก็มีค่ามาก — ได้เห็นภาพรวมว่าเปลี่ยนอะไรไปบ้าง จับไฟล์ที่เผลอแก้/เผลอใส่ความลับได้ และเป็นการฝึกสายตาแบบมืออาชีพ.

- **merge คืออะไร:** การ "รวม" สองเส้นเวลาเข้าด้วยกัน. ส่วนใหญ่ Git รวมให้เองอัตโนมัติ.

- **merge conflict คืออะไร (จากศูนย์):** ถ้า **บรรทัดเดียวกันของไฟล์เดียวกัน** ถูกแก้ไปคนละแบบบนสองเส้นเวลา Git จะ "ตัดสินใจแทนเราไม่ได้" ว่าจะเอาเวอร์ชันไหน — มันเลยหยุดแล้วขอให้ "คนตัดสิน". นี่ไม่ใช่ error ที่น่ากลัว มันคือ Git บอกว่า "ตรงนี้นายต้องเลือกเอง". เราจะ **จงใจสร้าง conflict** ในไฟล์ซ้อมเพื่อให้ผู้เรียนเจอในที่ปลอดภัย แล้วฝึกแก้จนหายกลัว.

- **`.gitignore` (เจาะลึก):** ไฟล์รายชื่อ "สิ่งที่ห้าม Git แตะ/เซฟ". เราจะอธิบายว่าทำไม **ห้าม** commit 3 อย่างนี้:
  - `node_modules/` — โฟลเดอร์ library ที่ติดตั้งจาก `package.json` มันใหญ่มาก (เป็นหมื่นไฟล์) และ "สร้างใหม่ได้เสมอ" ด้วยการ `npm install`. commit ขึ้นไปคือทำ repo บวมเปล่า ๆ.
  - `.env` — ไฟล์เก็บ **ความลับ** (รหัส, key, connection string). ถ้าหลุดขึ้น GitHub = ความลับรั่วสู่สาธารณะ (โดยเฉพาะ Phase 4–5 ที่จะมีของจริง). **เน้นย้ำหนัก ๆ.**
  - `dev.db` — ไฟล์ฐานข้อมูล SQLite (จะเกิดใน Phase 4) เป็นข้อมูล local เฉพาะเครื่องเรา ไม่ควรปนขึ้น repo.

> 📦 **กล่องเสริม: branch / PR / merge ต่างกันยังไง (สรุปสั้น)**
> - **branch** = เส้นเวลาทดลอง (เรื่องของ Git บนเครื่องเรา + บน GitHub)
> - **push branch** = อัปโหลดเส้นเวลานั้นขึ้น GitHub ให้คนอื่นเห็น
> - **Pull Request** = "หน้าเว็บขออนุมัติรวม" บน GitHub (ดู diff + คุยกัน)
> - **merge** = กดรวมจริง — `main` เปลี่ยนหลังจากนี้
> - **pull** = ดึงของล่าสุด (ที่ merge เข้าไปแล้ว) จาก GitHub กลับลงเครื่องเรา
>
> วงจรครบหนึ่งรอบคือ: **branch → commit → push → PR → review → merge → กลับ main → pull**. จำลำดับนี้ไว้ เดี๋ยวเราเดินตามมันเป๊ะ ๆ.

### 👣 ลำดับขั้นจับมือทำ

> **ก่อนเริ่ม:** เปิด integrated terminal ใน VS Code ที่โฟลเดอร์ `taskflow`. ตรวจให้แน่ใจว่างาน Phase 3 commit + push ครบแล้ว (terminal สะอาด ไม่มีของค้าง) เพื่อให้เราเริ่มจากจุดที่ `main` เรียบร้อย.

**— ตอนที่ 1: ทำความรู้จัก branch —**

1. **ดูว่าตอนนี้เรายืนอยู่ branch ไหน:** ให้พิมพ์
   ```bash
   git branch
   ```
   จะเห็นรายชื่อ branch ทั้งหมด มีดอกจัน `*` หน้าตัวที่กำลังยืนอยู่ (น่าจะเป็น `main`). อธิบาย: ดอกจัน = "คุณอยู่ตรงนี้".
   > 💡 ถ้าพิมพ์แล้วมันค้าง/เปิดหน้าเลื่อนดู ให้กด `q` เพื่อออก (Git ชอบเปิด pager). เป็นเรื่องปกติ ไม่ได้พัง.

2. **สร้าง branch ใหม่สำหรับ "ฟีเจอร์" แรก:** เราจะลองเพิ่มไฟล์ note เล็ก ๆ เป็นฟีเจอร์ซ้อม. พิมพ์
   ```bash
   git switch -c docs/notes
   ```
   อธิบายทีละชิ้น: `git switch` = สลับ branch, `-c` = create (สร้างใหม่ไปด้วยเลย), `docs/notes` = ชื่อ branch ที่เราตั้ง. **ชื่อ branch ควรสื่อว่าทำอะไร** (เราตั้งแนว `ประเภท/เรื่อง` เช่น `docs/...`, `feat/...`, `fix/...`). พิมพ์ `git branch` อีกรอบให้เห็นว่าดอกจันย้ายมาที่ `docs/notes` แล้ว — "เราย้ายจักรวาลมาแล้ว".

   > 🔎 **เช็กความเข้าใจสั้น ๆ:** ถามผู้เรียน — "ตอนนี้ commit ใหม่จะลงไปที่ branch ไหน?" (ตอบ: `docs/notes` เพราะดอกจันอยู่ตรงนั้น).

3. **แก้ของบน branch นี้ (commit บน branch):** สร้างไฟล์ `NOTES.md` ในโฟลเดอร์โปรเจกต์ (ใช้ VS Code สร้างไฟล์ใหม่ก็ได้) ใส่ข้อความสักบรรทัด เช่น:
   ```
   # บันทึกการเรียน TaskFlow
   - เริ่มฝึกใช้ branch กับ Pull Request แล้ว
   ```
   แล้ว save. กลับมา terminal:
   ```bash
   git status
   git add NOTES.md
   git commit -m "docs: add learning notes file"
   ```
   อธิบาย: นี่คือ commit เดิมที่เคยทำทุก Phase **แต่คราวนี้มันลงบน `docs/notes` ไม่ใช่ `main`**. ลองพิสูจน์: สลับกลับ `main` ชั่วคราว
   ```bash
   git switch main
   ```
   เปิดดูโฟลเดอร์ — **`NOTES.md` หายไป!** เพราะมันอยู่อีกจักรวาลหนึ่ง. แล้วสลับกลับ
   ```bash
   git switch docs/notes
   ```
   ไฟล์กลับมา. **โมเมนต์ "อ๋อ": branch แยกของจริงออกจากกันได้.**

4. **push branch ขึ้น GitHub:** ที่ผ่านมาเรา push `main`. คราวนี้ push branch ใหม่:
   ```bash
   git push -u origin docs/notes
   ```
   อธิบาย: `origin` = ชื่อเล่นของ repo บน GitHub, `-u` = ตั้งให้ branch local ตัวนี้ "ผูก" กับ branch บน GitHub (ครั้งหน้าพิมพ์แค่ `git push` พอ). เปิดหน้า repo บน GitHub — จะเห็นกล่องเหลือง ๆ ขึ้นมาว่า `docs/notes had recent pushes` พร้อมปุ่มเขียว **"Compare & pull request"**. นั่นแหละ ประตูสู่ขั้นต่อไป.

**— ตอนที่ 2: เปิด Pull Request และ merge (วิธีเว็บ GitHub) —**

5. **เปิด Pull Request:** กดปุ่มเขียว **"Compare & pull request"**. GitHub จะพาไปหน้าเปิด PR:
   - ช่องบนสุดบอกว่า **รวมจาก `docs/notes` → เข้า `main`** (ทิศทางสำคัญ: "เอาของจาก branch เรา เข้าหลัก").
   - ใส่ **title** สั้น ๆ (ดึงจาก commit ให้แล้ว แก้ได้) และ **description** บอกว่าทำอะไร เช่น "เพิ่มไฟล์ NOTES.md ไว้บันทึกการเรียน".
   - กด **"Create pull request"**.
   อธิบาย: ตอนนี้ยัง **ไม่ได้รวม** อะไรเลย — เราแค่ "เปิดคำขอ" ไว้. `main` ยังเหมือนเดิมเป๊ะ.

6. **review PR ของตัวเอง:** ในหน้า PR กดแท็บ **"Files changed"**. นี่คือ **diff** — สีเขียวคือบรรทัดที่เพิ่ม, สีแดงคือที่ลบ. พาผู้เรียนอ่าน: "เราเพิ่มไฟล์ `NOTES.md` 2 บรรทัด ถูกต้องไหม? มีไฟล์อื่นหลุดเข้ามาด้วยหรือเปล่า?". ย้ำว่า **นิสัยอ่าน diff ก่อน merge นี้แหละคือสิ่งที่แยกมือใหม่กับมืออาชีพ** — งานจริงคนในทีมจะมาอ่านตรงนี้แล้วคอมเมนต์.

7. **merge PR:** กลับไปแท็บ **"Conversation"** กดปุ่มเขียว **"Merge pull request"** → **"Confirm merge"**. ตอนนี้เอง `main` บน GitHub ถึงจะมี `NOTES.md`. GitHub จะเสนอปุ่ม **"Delete branch"** — กดลบได้เลย (branch ทำงานเสร็จแล้ว ไม่ต้องเก็บ ลบบน GitHub ไม่กระทบ branch บนเครื่องเรา).

8. **กลับมา `main` แล้ว `pull`:** **สำคัญมากและมือใหม่ลืมบ่อย** — ตอนนี้ `main` บน GitHub ใหม่กว่า `main` บนเครื่องเรา (เพราะ merge เกิดบนเว็บ). ต้องดึงลงมา:
   ```bash
   git switch main
   git pull
   ```
   เปิดโฟลเดอร์ดู — ตอนนี้ `NOTES.md` อยู่บน `main` บนเครื่องเราแล้ว. **ครบวงจรหนึ่งรอบ!** (branch → commit → push → PR → review → merge → กลับ main → pull). ลบ branch local ที่ใช้เสร็จแล้วด้วยก็ได้:
   ```bash
   git branch -d docs/notes
   ```

> 🔁 **ทบทวนวงจร ก่อนไปต่อ:** ให้ผู้เรียนเล่าวงจร 8 ขั้นด้วยปากตัวเอง 1 รอบ. ถ้ายังติด ๆ ขัด ๆ ให้ทำซ้ำอีก 1 รอบกับไฟล์อื่น (เช่น แก้ 1 บรรทัดใน `NOTES.md`) ก่อนไปเรียน conflict.

**— ตอนที่ 3: ทางลัดด้วย `gh` CLI (ทำสิ่งเดียวกันโดยไม่ออกจาก terminal) —**

9. **ทำวงจรเดิมด้วย `gh`:** `gh` คือเครื่องมือ command line ของ GitHub (ติดตั้งไว้แล้วใน `SETUP.md`). มันทำสิ่งที่เราเพิ่งคลิกบนเว็บได้จาก terminal เลย. ลองรอบที่สองแบบนี้ — สร้าง branch ใหม่ แก้ `NOTES.md` เพิ่มอีกบรรทัด commit แล้ว:
   ```bash
   git switch -c docs/notes-more
   # (แก้ NOTES.md เพิ่ม 1 บรรทัด แล้ว save)
   git add NOTES.md
   git commit -m "docs: add second note"
   git push -u origin docs/notes-more
   gh pr create --fill
   gh pr view --web
   ```
   อธิบาย: `gh pr create --fill` = เปิด PR โดยดึง title/description จาก commit ให้อัตโนมัติ; `gh pr view --web` = เปิด PR นั้นในเบราว์เซอร์ให้ดู diff. เมื่อพอใจ merge จาก terminal ได้เลย:
   ```bash
   gh pr merge --merge --delete-branch
   git switch main
   git pull
   ```
   > 💡 ถ้า `gh` ถามให้ login ครั้งแรก ให้ดูขั้นตอน `gh auth login` ใน `SETUP.md`. ผู้สอนเช็คคำสั่ง `gh` ล่าสุดจาก `SETUP.md` ก่อนพาพิมพ์เสมอ. **ไม่ต้องท่องจำ flag — ทั้งวิธีเว็บและ `gh` ได้ผลเหมือนกัน เลือกอันที่ถนัด.**

**— ตอนที่ 4: จงใจสร้าง merge conflict แล้วพาแก้ (หัวใจของ mini-phase นี้) —**

> ⚠️ **อ่านก่อน:** ขั้นนี้เราจะ "ทำให้ Git งง" โดยตั้งใจ เพื่อให้ผู้เรียนเจอ conflict ในที่ปลอดภัย (ไฟล์ note ไม่ใช่โค้ดแอป). **อย่าข้าม** — conflict คือสิ่งที่ทุกคนต้องเจอในงานจริง และการเจอครั้งแรกแบบมีคนพาจะลดความกลัวมหาศาล.

10. **สร้างสองเส้นเวลาที่แก้บรรทัดเดียวกัน:** ตอนนี้อยู่ `main` ที่ sync แล้ว.
    1. เปิด `NOTES.md` แก้ **บรรทัดแรก** (เช่นบรรทัด `# บันทึกการเรียน TaskFlow`) ให้เป็นข้อความ A เช่น `# สมุดบันทึก TaskFlow (เวอร์ชัน A)` → save → commit ลง `main` โดยตรง (รอบนี้อนุโลมเพื่อสร้างสถานการณ์):
       ```bash
       git add NOTES.md
       git commit -m "docs: edit title (A)"
       ```
    2. ทีนี้สร้าง branch จาก commit *ก่อนหน้า* เพื่อให้มันไม่เห็นการแก้ A. ทำง่าย ๆ คือย้อน 1 step แล้วแตก branch:
       ```bash
       git switch -c docs/title-b HEAD~1
       ```
       อธิบาย: `HEAD~1` = "commit ก่อนหน้า 1 ขั้น" — เราแตก branch `docs/title-b` จากจุด *ก่อน* ที่จะแก้ A. บน branch นี้ เปิด `NOTES.md` แก้ **บรรทัดแรกเดียวกัน** ให้เป็นข้อความ B เช่น `# สมุดงาน TaskFlow (เวอร์ชัน B)` → save → commit:
       ```bash
       git add NOTES.md
       git commit -m "docs: edit title (B)"
       ```
    ตอนนี้เรามี **บรรทัดแรกเดียวกัน ถูกแก้เป็น A บน `main` และ B บน `docs/title-b`** — เชื้อเพลิงของ conflict พร้อมแล้ว.

11. **ทำให้ conflict โผล่:** กลับไป `main` แล้วลองรวม `docs/title-b` เข้ามา:
    ```bash
    git switch main
    git merge docs/title-b
    ```
    Git จะหยุดแล้วฟ้องประมาณว่า `CONFLICT (content): Merge conflict in NOTES.md ... Automatic merge failed`. **อธิบายให้ผู้เรียนสบายใจ: นี่ไม่ใช่ของพัง — Git แค่บอกว่า "บรรทัดนี้ฉันเลือกแทนนายไม่ได้ ช่วยเลือกที".** `git status` จะบอกว่าไฟล์ไหนติด conflict (ตรงนี้คือ `NOTES.md`).

12. **อ่านเครื่องหมาย conflict ให้ออก:** เปิด `NOTES.md` ใน VS Code จะเห็นหน้าตาแบบนี้:
    ```
    <<<<<<< HEAD
    # สมุดบันทึก TaskFlow (เวอร์ชัน A)
    =======
    # สมุดงาน TaskFlow (เวอร์ชัน B)
    >>>>>>> docs/title-b
    ```
    อธิบายเครื่องหมายทีละชิ้น (จากศูนย์):
    - `<<<<<<< HEAD` = "ตั้งแต่บรรทัดนี้ลงไปคือเวอร์ชันของฝั่งที่ฉันยืนอยู่ (`HEAD` = `main` = เวอร์ชัน A)"
    - `=======` = "เส้นแบ่งกลาง" — ข้างบนคือฝั่งเรา ข้างล่างคือฝั่งที่กำลังรวมเข้ามา
    - `>>>>>>> docs/title-b` = "จบเวอร์ชันของฝั่งที่รวมเข้ามา (B)"
    หน้าที่เรา: **ลบเครื่องหมายทั้งหมดทิ้ง แล้วเหลือไว้เฉพาะข้อความที่เราต้องการจริง ๆ** (อาจเลือก A, เลือก B, หรือผสมเองก็ได้).
    > 💡 VS Code มักขึ้นปุ่มเล็ก ๆ เหนือ conflict ว่า **Accept Current Change / Accept Incoming Change / Accept Both**. ใช้ปุ่มก็ได้ แต่ **รอบแรกให้ลบเครื่องหมายด้วยมือ** เพื่อให้เข้าใจว่าปุ่มไปทำอะไร.

13. **แก้ให้จบ:** สมมติเราเลือกผสมเป็นบรรทัดเดียว แก้ไฟล์ให้เหลือ (ลบ `<<<`, `===`, `>>>` ออกหมด):
    ```
    # สมุดบันทึกงาน TaskFlow
    ```
    save. แล้วบอก Git ว่า "แก้เสร็จแล้ว":
    ```bash
    git add NOTES.md
    git commit
    ```
    (Git จะเปิดข้อความ merge commit มาให้ — กด save/ปิด editor ตามปกติ หรือถ้าเปิด `vim` ให้พิมพ์ `:wq` แล้ว Enter; รายละเอียดการตั้ง editor ดู `SETUP.md`). พิมพ์ `git status` ยืนยันว่าสะอาดแล้ว. **ผู้เรียนเพิ่งแก้ merge conflict ครั้งแรกสำเร็จ — ย้ำว่านี่คือทักษะที่หลายคนกลัวเกินเหตุ จริง ๆ แค่ "เลือกบรรทัดที่อยากเก็บ แล้วลบเครื่องหมาย".**

    > 🧯 **ปุ่มหนีฉุกเฉิน (สอนไว้ให้อุ่นใจ):** ถ้าระหว่าง conflict รู้สึกงงจนอยากเริ่มใหม่ พิมพ์ `git merge --abort` จะ **ยกเลิก merge กลับไปจุดก่อนชนทุกอย่าง** เหมือนไม่เคยกด merge. มี undo เสมอ ไม่ต้องกลัว.

14. **เก็บกวาด:** ลบ branch ซ้อมที่ไม่ใช้ และ push `main` ที่แก้ conflict แล้วขึ้น GitHub:
    ```bash
    git branch -d docs/title-b
    git push
    ```

**— ตอนที่ 5: เจาะลึก `.gitignore` (เตรียมตัวก่อน Phase 4) —**

15. **เปิดดู `.gitignore` ที่มีอยู่:** Next.js สร้างไฟล์ `.gitignore` ให้ตั้งแต่ Phase 0. เปิดดูด้วยกัน. อธิบายว่าแต่ละบรรทัดคือ "pattern ของสิ่งที่ Git จะแกล้งมองไม่เห็น" — มันจะไม่โผล่ใน `git status` และ `git add .` จะไม่หยิบมันไป.

16. **พิสูจน์ว่า `node_modules/` ถูก ignore จริง:** พิมพ์
    ```bash
    git status
    ```
    ชี้ให้เห็นว่า **ไม่มี** `node_modules/` โผล่มาเลย ทั้งที่ในโฟลเดอร์มีไฟล์เป็นหมื่น — เพราะ `.gitignore` กันไว้. อธิบายเหตุผลย้ำ (สร้างใหม่ได้ด้วย `npm install`, ใหญ่เกินจะ commit).

17. **เตรียม pattern สำหรับ Phase 4–5:** ให้ผู้เรียนตรวจว่า `.gitignore` มี (หรือเพิ่มถ้ายังไม่มี) บรรทัดเหล่านี้ เพื่อกัน **ความลับ** และ **ฐานข้อมูล local** ที่จะเกิดในเฟสถัด ๆ ไป:
    ```gitignore
    # secrets — ห้ามหลุดขึ้น GitHub เด็ดขาด
    .env
    .env*.local

    # local database (Phase 4 จะเกิด dev.db)
    *.db
    *.db-journal
    ```
    อธิบาย **ทำไมเรื่องนี้ใหญ่กว่า node_modules:** `.env` มีรหัส/ key จริง — ถ้าหลุดขึ้น public repo คนทั้งโลกเห็น และเปลี่ยนรหัสทีหลังก็ยังถือว่าเคยรั่ว. **กฎเหล็ก: ความลับไม่เคยขึ้น git.** (Phase 7 จะสอนทำ `.env.example` ที่ commit ได้ — ไฟล์ตัวอย่างที่ไม่มีค่าจริง — แต่ตอนนี้แค่รู้ว่า "`.env` จริงห้าม commit" ก็พอ.)

    > ⚠️ **ถ้าเผลอ commit ความลับไปแล้วจะทำยังไง?** การลบไฟล์แล้ว commit ใหม่ **ไม่พอ** เพราะมันยังอยู่ในประวัติ. ตอนนี้ผู้เรียนยังไม่มีความลับจริง (ยังไม่ถึง Phase 4) เลยปลอดภัย — แต่ให้จำหลักไว้ว่า "ถ้าเคยหลุด ให้ถือว่าความลับนั้นใช้ไม่ได้แล้ว ต้องไปสร้าง/เปลี่ยนค่าใหม่ทันที". รายละเอียดการกู้ประวัติเป็นเรื่องระดับสูง ไว้ค่อยเรียน.

**— ตอนที่ 6: รับกติกาทำงานต่อ —**

18. **ประกาศกติกา "1 ฟีเจอร์ = 1 branch = 1 PR":** สรุปกับผู้เรียนว่า **ตั้งแต่ Phase 4 เป็นต้นไป ขอให้ทำทุกเฟส (หรือทุกฟีเจอร์ย่อยใหญ่ ๆ) บน branch ของมันเอง แล้วเปิด PR ก่อน merge เข้า `main`** แทนการ commit ลง `main` ตรง ๆ แบบเดิม. ตั้งชื่อ branch ตามแนว:
    - `feat/...` สำหรับฟีเจอร์ใหม่ (เช่น `feat/prisma-database`, `feat/auth`)
    - `fix/...` สำหรับแก้ bug
    - `docs/...` / `chore/...` สำหรับเอกสาร/งานบ้าน
    เช่น ก่อนเริ่ม Phase 4 ให้ขึ้น `git switch -c feat/database` แล้วทำงานทั้งเฟสบนนั้น จบเฟสค่อยเปิด PR + merge. **commit message ตอนจบเฟสยังใช้ข้อความเดิมที่ `CURRICULUM.md` ให้ไว้ทุกอย่าง — แค่ตอนนี้มันไปอยู่บน branch แล้วผ่าน PR เข้า `main`.**
    > 💡 ทำคนเดียวก็ยังคุ้ม: ได้ history ที่อ่านรู้เรื่อง (1 PR = 1 ก้อนงานที่เล่าได้), ได้ฝึก review ตัวเอง, และถ้าเฟสไหนทำพังก็ทิ้ง branch ได้โดย `main` ยังดีอยู่.

### ✅ Checkpoint/แบบฝึกหัด
- สร้าง branch ใหม่, commit บน branch นั้น, push, เปิด PR, อ่าน diff ใน "Files changed", merge, แล้ว `git switch main && git pull` ได้ครบวงจร **โดยไม่ต้องดูคู่มือ**.
- อธิบายเป็นคำพูดได้ว่า **"ทำไมทีมไม่ commit ลง `main` ตรง ๆ"** และ **"Pull Request มีไว้ทำอะไร"**.
- เจอ merge conflict (จากขั้นจงใจสร้าง) แล้ว **ชี้ได้ว่า `<<<<<<<`, `=======`, `>>>>>>>` แต่ละอันหมายถึงอะไร** และแก้จน commit สำเร็จ.
- เปิด `.gitignore` แล้วอธิบายได้ว่า **ทำไม `node_modules/`, `.env`, `dev.db` ห้าม commit** (โดยเฉพาะเหตุผลของ `.env` ที่ต่างจากอีกสองอัน).
- ทำวงจร branch→PR→merge ซ้ำได้อีก 1 รอบด้วย **`gh` CLI** (หรืออธิบายได้ว่า `gh pr create` / `gh pr merge` แทนการคลิกบนเว็บขั้นไหน).
- **แบบฝึกหัด:** ตั้งใจสร้าง conflict ขึ้นมาเองอีก 1 รอบกับ `NOTES.md` (แก้บรรทัดเดียวกันบนสอง branch) แล้วแก้ให้จบด้วยตัวเอง — ทำซ้ำจนรู้สึกว่า "เฉย ๆ แล้ว ไม่กลัว".
- **แบบฝึกหัด (เตรียม Phase 4):** ขึ้น branch ใหม่ชื่อ `feat/database` ไว้ล่วงหน้า (ยังไม่ต้องทำอะไรบนนั้น) เพื่อพร้อมใช้กติกา "1 ฟีเจอร์ = 1 branch = 1 PR" ทันทีที่เริ่ม Phase 4.

> 🚦 **ยังไม่ผ่านอย่าเพิ่งไป Phase 4:** ถ้าผู้เรียนยังสับสนวงจร branch→PR→merge หรือยังกลัว conflict ให้วนทำ "ตอนที่ 1–2" และ "ตอนที่ 4" ซ้ำกับไฟล์ซ้อมจนคล่อง. ความคล่องตรงนี้จะทำให้ทั้ง Phase 4–8 ราบรื่นขึ้นมาก.

### 💾 git commit
mini-phase นี้ตัวงานหลักคือ "ฝึกกระบวนการ" ซึ่งถูก merge เข้า `main` ผ่าน PR ไปแล้วระหว่างทาง (ไฟล์ `NOTES.md` + `.gitignore` ที่ปรับ). ปิดท้ายด้วยการยืนยันว่า `.gitignore` ที่อัปเดตถูก commit เรียบร้อย — ทำผ่าน branch + PR เพื่อฝึกกติกาใหม่:

```bash
git switch -c chore/gitignore-hardening
# (ยืนยัน .gitignore มี .env / .env*.local / *.db ตามตอนที่ 5)
git add .gitignore
git commit -m "chore: harden .gitignore for env secrets and local database"
git push -u origin chore/gitignore-hardening
gh pr create --fill   # หรือเปิด PR ผ่านหน้าเว็บ GitHub
# review diff → merge → กลับ main → pull
```

ข้อความ commit หลักของ mini-phase นี้:
```
chore: harden .gitignore for env secrets and local database
```

### ⏱️ ประเมินเวลา
ประมาณ 3–5 ชั่วโมง (เป็น mini-phase สั้น เน้นทำซ้ำให้ชิน — ส่วนที่กินเวลาสุดคือ "ตอนที่ 4: merge conflict" ซึ่งควรให้เวลาเต็มที่และทำซ้ำหลายรอบ ไม่ต้องรีบ). คุ้มมากเพราะกติกานี้จะใช้ยาวตลอด Phase 4–8.

## Phase 4 — Database & Backend

### 🎯 เป้าหมายของเฟส
ก้าวสู่ "งานจริงระดับมืออาชีพ": เก็บ task ใน **database จริง** แทน localStorage ผ่าน Prisma + SQLite และเขียน backend ของตัวเองด้วย **Server Actions**. เริ่มแนะนำ TypeScript ทีละนิดเพื่อความปลอดภัยของข้อมูล. นี่คือจุดที่ frontend และ backend มาเจอกัน.

> ⚠️ **เฟสนี้หนักที่สุดของหลักสูตร — มีของใหม่เยอะมาก (database, Prisma, schema, migration, Server Actions, TypeScript).** ห้ามรวบทำรวดเดียว. หลักสูตรแบ่งเฟสนี้เป็น **2 ช่วงย่อย** ให้พักหายใจตรงกลาง:
> - **Phase 4a — Database + อ่านข้อมูล:** ติดตั้ง Prisma, เขียน schema, migration, แล้วทำหน้าให้ **อ่าน** task จาก DB มาแสดง (ยังไม่เขียนข้อมูล). จบ 4a ควร commit 1 ครั้งและให้ผู้เรียนพัก.
> - **Phase 4b — เขียนข้อมูลด้วย Server Actions + TypeScript:** เพิ่ม/แก้ไข/ติ๊ก/ลบ task ลง DB ผ่าน Server Actions, ถอด localStorage, แล้วค่อยเริ่มใส่ TypeScript ทีละนิด.
>
> ถ้าผู้เรียนล้าให้หยุดจบที่ 4a ก่อนได้ — แอปยังรันได้ (อ่านข้อมูลจาก DB ขึ้นจริง) เป็นจุดพักที่ดี.

### 🧱 สิ่งที่จะสร้างเพิ่มใน TaskFlow
- Database จริง (SQLite ตอน dev) ที่มีตาราง `Task`
- Server Actions สำหรับ create/read/update/delete task ลง DB
- TaskFlow อ่าน-เขียนข้อมูลจาก DB แทน localStorage
- เริ่มมีไฟล์ `.ts`/type เบื้องต้น

### 📚 คอนเซปต์ที่จะสอน (พร้อมคำอธิบายสั้นๆ)
- **Database คืออะไร:** ที่เก็บข้อมูลถาวรแบบมีโครงสร้าง; อธิบายตาราง/แถว/คอลัมน์ เทียบกับ spreadsheet.
- **SQLite vs Postgres (ตอนนี้ใช้ SQLite):** SQLite = ไฟล์เดียว ตั้งค่าง่าย เหมาะ dev; Postgres = ตัวจริงสำหรับ production (จะย้ายตอน Phase 8).
- **Prisma & ORM:** Prisma ช่วยให้คุยกับ DB ด้วยโค้ด JavaScript/TypeScript แทนการเขียน SQL ดิบ; ORM = ตัวแปลงระหว่าง object ในโค้ดกับตารางใน DB.
- **Schema & migration:** `schema.prisma` คือพิมพ์เขียวของข้อมูล; migration = การเปลี่ยนโครงสร้าง DB อย่างมีประวัติ (เหมือน commit ของ DB).
- **Server Actions:** ฟังก์ชันที่รันบนเซิร์ฟเวอร์ เรียกจากฟอร์ม/ปุ่มได้ตรงๆ เพื่ออ่าน-เขียน DB อย่างปลอดภัย (**วิธีหลักและวิธีเดียวที่หลักสูตรนี้ใช้เขียนข้อมูล**). ดูกล่อง "โค้ดบรรทัดไหนรันที่ไหน" ด้านล่างเพื่อเข้าใจว่ามันรันที่ server ไม่ใช่เบราว์เซอร์.
- **TypeScript เบื้องต้น:** การบอก "ชนิดข้อมูล" (string, number, boolean) ให้โค้ด ช่วยจับ bug ก่อนรัน. เริ่มจากใส่ type ให้ props/ฟังก์ชันทีละนิด (ทำใน 4b หลัง backend นิ่งแล้ว ไม่ทำพร้อมกัน).
- **revalidate/refresh ข้อมูล:** ทำไมหลังเขียน DB ต้องบอก Next.js ให้โหลดข้อมูลใหม่ให้หน้าอัปเดต (เช่น `revalidatePath`).
- **(ข้ามไปก่อน) Route Handlers / API route:** หลักสูตรนี้ **ยังไม่ใช้** Route Handlers ในเฟสนี้ เพราะ Server Actions ครอบคลุมทุกอย่างที่ TaskFlow ต้องใช้แล้ว — การเกริ่นทั้งสองอย่างพร้อมกันจะทำให้สับสนว่าจะเขียนข้อมูลด้วยอะไร. **ผู้สอนไม่ต้องสอน Route Handlers ตอนนี้** (ถ้าผู้เรียนถามค่อยบอกสั้นๆ ว่า "เป็นอีกวิธีทำ API ไว้เรียนทีหลังตอนต้องเชื่อมกับแอปภายนอก").

> 🗺️ **กล่องสำคัญ: โค้ดบรรทัดไหนรันที่ server บรรทัดไหนรันที่ browser?** (มือใหม่สับสนจุดนี้บ่อยสุดในเฟสนี้)
> ทบทวนจาก Phase 3: ค่าเริ่มต้นของไฟล์ใน `app/` คือ **Server Component** (รันที่ server). จะกลายเป็น **Client Component** (รันที่ browser) ก็ต่อเมื่อมี `"use client"` บนสุดของไฟล์.
> | โค้ดส่วนนี้ | รันที่ไหน | เห็น `console.log` ที่ไหน | แตะ DB / Prisma ได้ไหม |
> |---|---|---|---|
> | Server Component (ไม่มี `"use client"`) — เช่นหน้า tasks ที่ดึงข้อมูล | **server** | terminal (ที่รัน `npm run dev`) | ✅ ได้ตรงๆ |
> | Server Action (ฟังก์ชันที่มี `"use server"`) | **server** | terminal | ✅ ได้ — ที่นี่คือที่เขียน DB |
> | Client Component (มี `"use client"`) — เช่น `AddTaskForm` ที่มี state | **browser** | console ในเบราว์เซอร์ (F12) | ⛔ ไม่ได้ — ต้อง "เรียก" Server Action แทน |
> วิธีพิสูจน์ให้ผู้เรียนเห็นจริง: ใส่ `console.log` ในหน้า tasks (server) แล้วดูว่าขึ้นใน **terminal** ไม่ใช่ในเบราว์เซอร์ — ตอกย้ำว่า "โค้ดนี้รันที่ server". `AddTaskForm` (มี `"use client"`) เป็นฝั่ง browser มันจึง **เรียก** Server Action ให้ไปทำงานที่ server ให้ แทนที่จะแตะ DB เอง.

### 👣 ลำดับขั้นจับมือทำ

**— ช่วง 4a: Database + อ่านข้อมูล (ทำให้แอป "อ่าน" จาก DB ได้ก่อน) —**
1. **ปูพื้น database ด้วยภาพ:** อธิบายตาราง `Task` (id, title, done, createdAt) ก่อนแตะโค้ด.
2. **ติดตั้ง & ตั้งค่า Prisma + SQLite** ตาม `SETUP.md` **§11**: install Prisma, `npx prisma init --datasource-provider sqlite`. (ผู้สอนเช็คคำสั่งล่าสุดที่ <https://www.prisma.io/docs> ก่อนเสมอ).
3. **เขียน schema แรก:** นิยาม model `Task` ใน `schema.prisma` อธิบายแต่ละ field.
4. **รัน migration แรก:** สร้างตารางจริงใน DB; เปิด Prisma Studio ให้เห็นตารางเปล่าๆ (เห็นภาพ DB จริง).
   > 💾 **เตือนเรื่อง backup (จุดแรกที่มี `.env` และ `dev.db`):** ตั้งแต่ตอนนี้โปรเจกต์มีไฟล์ `.env` (ความลับ) กับ `dev.db` (ฐานข้อมูล local) เกิดขึ้น — **ทั้งสองไฟล์นี้ถูก `.gitignore` กันไว้ จึงไม่ขึ้น git/GitHub** (ตามที่ฝึกใน Mini-Phase B). แปลว่า `git push` ไม่ได้ backup สองไฟล์นี้ให้: ถ้าเครื่องพังหรือเผลอลบ มันหายเลย กู้จาก GitHub ไม่ได้. ตอนนี้ยังไม่ต้องทำอะไรมาก แค่รู้ไว้ว่า "ค่าใน `.env` ต้องจดเก็บที่ปลอดภัยไว้เอง" และ "ข้อมูลใน `dev.db` เป็นแค่ข้อมูลทดสอบ local สร้างใหม่ได้". เรื่อง backup จริงจัง + การตั้ง env บน cloud จะย้ำอีกครั้งตอน Phase 8 (Deploy).
5. **สร้าง Prisma client + ฟังก์ชันอ่าน task:** ทำหน้า tasks ให้ดึงข้อมูลจาก DB (เป็น **Server Component** อ่าน DB ตรงๆ — ทบทวนกล่อง "โค้ดบรรทัดไหนรันที่ไหน") — เพิ่มข้อมูลทดสอบผ่าน Prisma Studio แล้วเห็นขึ้นหน้าเว็บ. **ทดลองใส่ `console.log` ในหน้านี้ แล้วชี้ให้เห็นว่าขึ้นใน terminal ไม่ใช่เบราว์เซอร์** (พิสูจน์ว่ารันที่ server).
   > ⏸️ **จุดพัก 4a:** ถึงตรงนี้แอปอ่านข้อมูลจาก DB ขึ้นจริงแล้ว — commit สักครั้ง (เช่น `feat: read tasks from database via Prisma`) แล้วให้ผู้เรียนพัก ก่อนเริ่ม 4b.

**— ช่วง 4b: เขียนข้อมูลด้วย Server Actions แล้วค่อยเพิ่ม TypeScript —**
6. **เขียน Server Action สำหรับ "เพิ่ม task":** สร้างฟังก์ชันที่มี `"use server"` (รันที่ server — ดูกล่องด้านบน) ผูกกับ `AddTaskForm` ให้บันทึกลง DB จริง แล้ว `revalidatePath` ให้หน้าอัปเดต. ย้ำว่า `AddTaskForm` อยู่ฝั่ง browser แต่ **เรียก** action ให้ไปทำงานที่ server.
7. **toggle done + แก้ไขชื่อ + ลบ ผ่าน Server Action:** ย้ายตรรกะ toggle/edit/delete จาก state ฝั่ง client มาเขียน DB. การแก้ไขชื่อ task ก็ทำผ่าน Server Action เช่นกัน (รับ id + title ใหม่ → `prisma.task.update(...)` → `revalidatePath`) — โหมด inline edit ฝั่ง UI ยังเหมือน Phase 2 แต่ตอนกด "บันทึก" ให้เรียก Server Action แทนการแก้ state ตรงๆ.
8. **ถอด localStorage ออก:** อธิบายว่าตอนนี้ "แหล่งความจริง" คือ DB แล้ว; ลบโค้ด localStorage ที่ไม่ใช้.
9. **เริ่มใส่ TypeScript ทีละนิด (ทำเป็นขั้นสุดท้าย หลัง backend ทำงานครบแล้ว):** เปลี่ยนไฟล์ที่เกี่ยวกับ task เป็น `.ts`/`.tsx`, ใส่ type ให้ Task และ props ของ `TaskItem` (Prisma มี type ให้ใช้อยู่แล้ว — ชี้ให้เห็นว่า type ช่วยเตือนตอนพิมพ์ผิด). ทำหลังสุดเพื่อไม่ให้ "ยากซ้อนยาก" ระหว่างกำลังงงเรื่อง server/client.

### ✅ Checkpoint/แบบฝึกหัด
- เพิ่ม/แก้ไขชื่อ/ติ๊ก/ลบ task แล้วเปิด Prisma Studio เห็นข้อมูลเปลี่ยนใน DB จริง.
- รีเฟรชหรือเปิดจากเบราว์เซอร์อื่นแล้วข้อมูลยังอยู่ (ต่างจาก localStorage).
- อธิบายได้ว่า Server Action ต่างจากฟังก์ชันฝั่ง client ยังไง.
- แบบฝึกหัด: เพิ่ม field ใหม่ใน schema (เช่น `priority`) แล้วรัน migration เอง 1 รอบ.

ประมาณ 14–20 ชั่วโมง (เฟสที่หนักสุดของหลักสูตร — มีของใหม่หลายชั้นพร้อม TypeScript). **แบ่งทำ 4a แล้วพัก ก่อนเริ่ม 4b** อย่าฝืนรวดเดียว.

## Phase 5 — Auth & Multi-user

### 🎯 เป้าหมายของเฟส
ทำให้ TaskFlow รองรับ "ผู้ใช้หลายคน" — สมัคร/ล็อกอินได้ และแต่ละคนเห็นเฉพาะ task ของตัวเอง. เข้าใจแนวคิด authentication/session และ security เบื้องต้น ซึ่งเป็นทักษะที่งานจริงต้องใช้แทบทุกโปรเจกต์.

### 🧱 สิ่งที่จะสร้างเพิ่มใน TaskFlow
- ระบบ signup/login ด้วย Auth.js (NextAuth)
- `User` model ใน DB และความสัมพันธ์ `User` ↔ `Task`
- task ผูกกับเจ้าของ — เห็นเฉพาะของตัวเอง
- protected route (หน้า tasks ต้องล็อกอินก่อน)

### 📚 คอนเซปต์ที่จะสอน (พร้อมคำอธิบายสั้นๆ)
- **Authentication vs Authorization:** authentication = "คุณเป็นใคร" (ล็อกอิน); authorization = "คุณมีสิทธิ์ทำอะไร" (เห็น task ของตัวเองเท่านั้น).
- **Session:** วิธีที่เซิร์ฟเวอร์ "จำ" ว่าใครล็อกอินอยู่ข้ามหลายหน้า.
- **Auth.js (NextAuth):** library จัดการ login/session ให้ ไม่ต้องเขียนเองทั้งหมด; หลักสูตรนี้ใช้ **Credentials provider** (อีเมล+รหัสผ่าน) เป็นหลัก — โครงไฟล์/คำสั่งติดตั้งอยู่ใน `SETUP.md` §12.
- **Relation ใน database:** ความสัมพันธ์ระหว่างตาราง (`User` มีหลาย `Task`) ด้วย foreign key; ปรับ schema Prisma ให้มี relation.
- **Protected route / middleware:** กันไม่ให้คนที่ยังไม่ล็อกอินเข้าหน้าบางหน้า; redirect ไปหน้า login. **บน Next.js 16 ไฟล์ middleware ถูกเปลี่ยนชื่อเป็น `proxy.ts`** (ไม่ใช่ `middleware.ts` แล้ว) — สำหรับ Auth.js v5 ให้ `export { auth as middleware }` ไว้ใน **`proxy.ts`**. ดูรายละเอียด + คำเตือนใน `SETUP.md` ข้อ 9.10.
- **Security เบื้องต้น:** ทำไมต้อง hash password (ไม่เก็บ plain text), อย่าวางความลับไว้ฝั่ง client, ตรวจสิทธิ์ที่ฝั่ง server เสมอ (ห้ามเชื่อ client).

### 👣 ลำดับขั้นจับมือทำ
1. **อธิบายภาพรวม auth ก่อนโค้ด:** flow ของ signup → login → session → ออกจากระบบ.
2. **ติดตั้ง & ตั้งค่า Auth.js** ตาม `SETUP.md` **§12** (install `next-auth@beta`, สร้าง `AUTH_SECRET` ด้วย `npx auth secret`, โครงไฟล์ `src/auth.ts` + route handler + `proxy.ts`) — อธิบายว่า env คือความลับที่ไม่ commit ขึ้น git. (ผู้สอนเช็ค <https://authjs.dev> ก่อนเสมอ).
3. **เพิ่ม `User` ลง schema + relation กับ `Task`:** รัน migration; อธิบายว่า task ต้องมี `userId`.

## Mini-Phase C — Async JavaScript + ดึง API ภายนอก (แทรกหลัง Phase 6)

> **mini-phase นี้คืออะไร / แทรกตรงไหน:** เป็นเฟสย่อย (mini-phase) ที่แทรก **หลังจากจบ Phase 6** — ตอนนี้ TaskFlow เป็นแอปเต็มแล้ว: มี auth (สมัคร/ล็อกอิน, เห็นเฉพาะ task ของตัวเอง), มี database จริง (Prisma + SQLite), เพิ่ม/แก้/ติ๊ก/ลบ task ผ่าน **Server Actions**, มีหมวด/due date/search/filter/dashboard/dark mode แล้ว. ทั้งหมดที่ผ่านมาเรา "อ่าน-เขียนข้อมูลของตัวเอง" บนเซิร์ฟเวอร์ของเราเอง. เฟสนี้จะเปิดโลกใหม่: **ดึงข้อมูลจากเซิร์ฟเวอร์ของคนอื่นบนอินเทอร์เน็ต** (เรียกว่า "external API") ซึ่งเป็นทักษะที่งานจริงเกือบทุกแอปต้องใช้ (เช่น ดึงพยากรณ์อากาศ, อัตราแลกเปลี่ยน, ข่าว, แผนที่).
>
> **สำหรับผู้สอน (อ่านก่อนเริ่ม):** เฟสนี้สั้นและ "เห็นผลไว" ตั้งใจให้เป็นช่วงพักสมองหลัง Phase 6 ที่หนัก แต่สอนคอนเซปต์สำคัญที่ Phase ก่อนหน้าจงใจยังไม่แตะ คือ **asynchronous JavaScript** (`Promise` / `async` / `await` / `fetch`). ห้ามรวบ — พาทำทีละ sub-step ใน "👣 ลำดับขั้นจับมือทำ" รันให้เห็นในเบราว์เซอร์บ่อย ๆ. คอนเซปต์ async เป็นเรื่อง "นามธรรม" สำหรับมือใหม่ ใช้คำเปรียบเทียบเยอะ ๆ และอย่ารีบ.

### 🎯 เป้าหมายของเฟส

ให้ผู้เรียนเข้าใจ "ทำไมโค้ดบางอย่างต้อง **รอ**" — นั่นคือแนวคิด **asynchronous** (อะซิงโครนัส) — แล้วใช้มันดึงข้อมูลสด ๆ จาก **เซิร์ฟเวอร์ของคนอื่นบนอินเทอร์เน็ต** ผ่าน `fetch()` มาแสดงใน TaskFlow ได้จริง. จบเฟสนี้ผู้เรียนต้องตอบได้ว่า "การดึงข้อมูลจาก external API ต่างจาก Server Actions ที่เราใช้มาตลอดยังไง" และจัดการสถานะ **กำลังโหลด (loading)** กับ **พัง/เน็ตช้า (error)** เป็น — เพราะของจริงบนอินเทอร์เน็ต "ไม่ตอบทันที และบางทีก็ล่ม".

เป้าหมายย่อยด้านความรู้สึก: ลดความกลัวคำว่า "async/await/Promise" ที่ผู้เรียนมักเจอแล้วงง — ทำให้รู้สึกว่า "อ๋อ มันก็แค่การรอแบบมีมารยาท".

### 🧱 สิ่งที่จะทำ/สร้างใน TaskFlow

- การ์ด **"คำคมให้กำลังใจประจำวัน" (Daily Quote)** บนหน้า **Dashboard** ที่ดึงข้อความสด ๆ จาก **public API ฟรี (ไม่ต้องใช้ API key)** ทุกครั้งที่เปิดหน้า
- การ์ดนี้แสดง **3 สถานะ** ครบ: กำลังโหลด (loading), สำเร็จ (แสดงคำคม), และพลาด (error + ปุ่ม "ลองใหม่")
- ปุ่ม **"สุ่มคำคมใหม่"** เพื่อเรียก API ซ้ำ — ฝึกเรียก async ซ้ำตามการกดของผู้ใช้
- (แบบฝึกหัดท้ายเฟส) ทางเลือก: การ์ด **"วันนี้วันอะไร / วันหยุดถัดไป"** ดึงจาก public holiday API เพื่อแนะนำ due date — ใช้คอนเซปต์เดิมทุกอย่าง

> ทั้งหมดนี้ทำด้วย **Client Component + `useEffect` + `fetch`** (เดี๋ยวอธิบายว่าทำไมใช้ฝั่ง client ตรงนี้ ทั้งที่ก่อนหน้านี้เราใช้ server เป็นหลัก).

### 📚 คอนเซปต์ที่จะสอน (อธิบายสั้น ๆ จากศูนย์)

- **Synchronous vs Asynchronous (หัวใจของเฟส):**
  - **Synchronous (ซิงโครนัส) = ทำทีละอย่าง ต้องเสร็จก่อนถึงทำบรรทัดต่อไป.** โค้ดส่วนใหญ่ที่ผ่านมาเป็นแบบนี้ เช่น `2 + 2` ได้ผลทันที.
  - **Asynchronous (อะซิงโครนัส) = สั่งงานที่ "ใช้เวลา" ไว้ แล้วไปทำอย่างอื่นต่อ ไม่ยืนค้างรอ.** พอผลลัพธ์มาถึงค่อยกลับมาจัดการ.
  - **เปรียบเทียบสั่งกาแฟ:** ไปร้านกาแฟ สั่งลาเต้ → ถ้าเป็นแบบ synchronous คุณต้อง **ยืนแช่อยู่ที่เคาน์เตอร์** ขยับไปไหนไม่ได้จนกาแฟเสร็จ (เสียเวลาเปล่า). แบบ asynchronous: บาริสต้าให้ **เบอร์คิว** แล้วบอก "เดี๋ยวเรียกนะ" → คุณไปนั่งเล่นมือถือ/คุยกับเพื่อนได้ → พอกาแฟเสร็จเขาเรียกเบอร์ คุณค่อยไปรับ. การดึงข้อมูลจากอินเทอร์เน็ต "ใช้เวลา" (เพราะข้อมูลเดินทางข้ามโลก) เราจึงไม่อยากให้หน้าเว็บค้างแข็งระหว่างรอ — เลยทำแบบ async.
- **`Promise` (พรอมิส) คืออะไร:** "เบอร์คิวกาแฟ" ในโลกโค้ด — เป็น **คำสัญญาว่าจะมีผลลัพธ์ในอนาคต** (อาจสำเร็จหรือพลาดก็ได้). ตอนเรียก `fetch()` มันยังไม่ได้คืนข้อมูลทันที แต่คืน Promise ("สัญญาว่าเดี๋ยวข้อมูลจะมา"). Promise มี 3 สถานะ: กำลังรอ (pending), สำเร็จ (fulfilled), ล้มเหลว (rejected) — ตรงกับ loading/success/error ที่เราจะทำพอดี.
- **`async` / `await`:** วิธีเขียน Promise ให้ "อ่านเหมือนโค้ดธรรมดาบรรทัดต่อบรรทัด" แทนที่จะซ้อนกันยุ่ง.
  - `async` วางหน้า function เพื่อบอกว่า "ฟังก์ชันนี้มีการรอ".
  - `await` วางหน้าการรอ แปลว่า "หยุดรอตรงนี้จนกว่าจะได้ผล แล้วค่อยไปบรรทัดถัดไป" (แต่หยุดเฉพาะ "งานนี้" ไม่ได้ทำให้ทั้งเบราว์เซอร์ค้าง). พูดง่าย ๆ คือ `await` = "รอเบอร์คิวนี้ให้เสร็จก่อนค่อยไปต่อ".
- **`fetch()` คืออะไร:** ฟังก์ชันมาตรฐานสำหรับ "ส่งคำขอไปเซิร์ฟเวอร์อื่นแล้วขอข้อมูลกลับมา" (เหมือนเบราว์เซอร์เปิด URL ให้เรา แต่ทำในโค้ด). มันคืน Promise เสมอ จึงคู่กับ `await`.
- **JSON คืออะไร / parse ยังไง:** **JSON** (อ่านว่า เจ-สัน) = รูปแบบข้อความมาตรฐานที่เซิร์ฟเวอร์ใช้ "ห่อข้อมูล" ส่งกลับมา หน้าตาเหมือน object ของ JavaScript (มี `{ "key": "value" }`). แต่ของที่ `fetch` คืนมาเป็น **ข้อความ (text)** ก่อน เราต้อง **แปลง (parse)** ให้กลายเป็น object ที่โค้ดใช้ได้ ด้วย `.json()` (ซึ่งก็เป็น async อีกตัว ต้อง `await`). เปรียบเทียบ: เซิร์ฟเวอร์ส่งของมาเป็น "พัสดุห่อปิด" — `.json()` คือการแกะกล่องออกมาเป็นของที่หยิบใช้ได้.
- **Loading state & Error state (จัดการตอนเน็ตช้า/พัง):** เพราะ external API "ไม่อยู่ในมือเรา" จึงต้องเผื่อ 2 กรณีที่ Server Actions ไม่ค่อยเจอ: (1) **ระหว่างรอ** ต้องโชว์ "กำลังโหลด..." ไม่ใช่หน้าว่างเปล่าให้ผู้ใช้งง; (2) **ถ้าพลาด** (เน็ตหลุด / API ล่ม / ตอบ error) ต้องโชว์ข้อความเป็นมิตร + ปุ่มลองใหม่ ไม่ใช่จอขาวหรือแอปพัง. ใช้ `try / catch` จับ error และใช้ `useState` เก็บสถานะ loading/error.
- **ทำไม external API ต่างจาก Server Actions (จุดที่ต้องตอกย้ำ):**

  | | **Server Actions (ที่เราใช้มาตลอด)** | **External API (เฟสนี้)** |
  |---|---|---|
  | คุยกับใคร | **เซิร์ฟเวอร์ของเราเอง** (DB ของ TaskFlow) | **เซิร์ฟเวอร์ของคนอื่น** บนอินเทอร์เน็ต |
  | ข้อมูลเป็นของใคร | ของเรา ควบคุมได้เต็ม | ของคนอื่น เราแค่ "ขอดู" |
  | เชื่อถือได้แค่ไหน | เรารู้ว่ามันทำงานยังไง | อาจช้า/ล่ม/เปลี่ยน/ปิดเมื่อไหร่ก็ได้ |
  | เขียนข้อมูลกลับได้ไหม | ได้ (เพราะเป็นของเรา) | ส่วนใหญ่แค่อ่าน (โดยเฉพาะ API ฟรี) |
  | ต้องเผื่อ error เน็ตไหม | น้อย (รันบน server เดียวกัน) | **ต้องเผื่อเสมอ** |

- **ทำไมเฟสนี้ใช้ Client Component + `useEffect` (ไม่ใช่ Server Action):** จงใจ — เพื่อให้ผู้เรียนเห็น "อีกวิธีดึงข้อมูล" ที่ทำงาน **ในเบราว์เซอร์ของผู้ใช้** และได้ฝึกจัดการ loading/error ด้วยตา (เห็นคำว่า "กำลังโหลด..." วาบขึ้นจริง ๆ). `useEffect` คือที่ที่เหมาะกับ "งานข้างเคียงหลัง render" เช่น การไปดึงข้อมูลตอนหน้าเปิดขึ้น (ทบทวนจาก Phase 2 ที่เคยใช้ `useEffect` คุยกับ localStorage).

> 🧠 **กล่องเสริม: "ทำงานในเบราว์เซอร์" กับ "เรื่อง CORS" (อ่านเผื่องง):**
> เพราะเรา `fetch` จากฝั่งเบราว์เซอร์ (client) บางเว็บ API จะมีกฎความปลอดภัยชื่อ **CORS** ที่อาจ "ไม่อนุญาต" ให้เว็บอื่นเรียกจากเบราว์เซอร์ตรง ๆ (จะเห็น error สีแดงใน console F12 พูดถึง CORS). **public API ฟรีที่เราจะเลือก ส่วนใหญ่เปิด CORS ให้อยู่แล้ว** จึงไม่เจอปัญหานี้. ตอนนี้รู้แค่ว่า "ถ้าเจอคำว่า CORS แดง ๆ แปลว่า API ตัวนั้นไม่ให้เรียกจากเบราว์เซอร์ ให้สลับไปใช้ API ตัวอื่นที่เปิด CORS" ก็พอ — ไม่ต้องเข้าใจลึกตอนนี้.

### 🌐 เลือก API + วิธีทดสอบ endpoint ก่อนเขียนโค้ด (สำคัญมาก — ทำก่อนข้อ 1 เสมอ)

> ⚠️ **อ่านก่อน:** ส่วนนี้เขียนแบบ **API-agnostic** (ไม่ผูกกับ API ตัวใดตัวหนึ่งตายตัว) เพราะ public API ฟรีบนอินเทอร์เน็ต **เปิด/ปิด/เปลี่ยนรูปแบบได้ตลอดเวลา**. หลักการสำคัญที่ผู้เรียนต้องจำติดตัว: **ก่อนเขียนโค้ด `fetch` ใด ๆ ให้ "เปิด URL ของ API นั้นในเบราว์เซอร์ดูด้วยตาก่อนเสมอ"** ว่ามันยังมีชีวิตและคืน JSON หน้าตายังไง.

**ตัวอย่างที่หลักสูตรนี้ใช้ (ณ วันที่เขียน):** public quote API ฟรีที่ไม่ต้องใช้ key เช่น **`https://zenquotes.io/api/random`** (คืน JSON เป็น array ที่มี `q` = ข้อความคำคม, `a` = ชื่อผู้พูด). ตัวสำรองที่ใช้คอนเซปต์เดียวกัน เช่น **`https://api.quotable.io/random`** (คืน object ที่มี `content` + `author`) หรือ **`https://dummyjson.com/quotes/random`** (คืน `quote` + `author`).

> 🔁 **หมายเหตุสำคัญ (ผู้สอนต้องอ่านออกเสียงให้ผู้เรียนฟัง):** **ถ้า API ตัวอย่างด้านบนเปลี่ยน / ปิด / เรียกไม่ขึ้น ให้สลับเป็น public API ฟรีตัวไหนก็ได้ที่ "ไม่ต้องใช้ key" และ "คืน JSON"** — โค้ดทั้งหมดในเฟสนี้ใช้ซ้ำได้ เปลี่ยนแค่ 2 จุด: (ก) **URL ใน `fetch`** และ (ข) **ชื่อ field ที่หยิบออกจาก JSON** (เช่นจาก `data[0].q` เป็น `data.content`). ไม่ต้องรื้อโครงสร้างใหม่. นี่คือเหตุผลที่เราต้อง "เปิดดู JSON ก่อน" — เพื่อรู้ว่าต้องหยิบ field ชื่ออะไร.

**👣 วิธีทดสอบ endpoint ก่อนเขียนโค้ด (ทำทุกครั้งกับ API ใหม่):**

1. **เปิด URL ตรง ๆ ในแท็บเบราว์เซอร์ใหม่.** เช่นวาง `https://zenquotes.io/api/random` ในช่อง address แล้ว Enter.
2. **ดูว่ามีข้อความ JSON ขึ้นไหม.** ถ้าเห็นข้อความหน้าตา `[{"q":"...","a":"..."}]` แปลว่า **API ยังมีชีวิตและคืน JSON** ✅. ถ้าเห็นหน้า error / 404 / ว่างเปล่า → API ตัวนี้ใช้ไม่ได้ตอนนี้ ให้สลับตัวอื่น.
3. **อ่านโครงสร้าง JSON ด้วยตา** ว่าข้อมูลที่เราอยากได้ "อยู่ field ชื่ออะไร" และ "ซ้อนกี่ชั้น" (เป็น array หรือ object?). จดไว้ — เดี๋ยวโค้ดต้องหยิบตรงนี้.
   - 💡 ทิป: ติดตั้ง extension จัดรูป JSON ในเบราว์เซอร์ หรือกด F12 → แท็บ Network เพื่อดู response สวย ๆ ได้ แต่แค่เปิด URL เฉย ๆ ก็พอเห็นแล้ว.
4. **(ทางเลือกขั้นโปร) ลองยิงด้วยเครื่องมือ:** ใช้ `curl "URL"` ใน terminal ก็เห็น JSON เหมือนกัน — แต่สำหรับมือใหม่ "เปิดในเบราว์เซอร์" เข้าใจง่ายสุด.

> 🧪 **Checkpoint ย่อยก่อนไปต่อ:** ผู้เรียนเปิด URL ของ API ในเบราว์เซอร์เองได้ และชี้ได้ว่า "ข้อความคำคมอยู่ตรง field ไหนของ JSON". ถ้ายังชี้ไม่ได้ อย่าเพิ่งไปเขียนโค้ด.

### 👣 ลำดับขั้นจับมือทำ

> ทำ "ทดสอบ endpoint" ในส่วนด้านบนให้ผ่านก่อนเริ่มข้อ 1.

1. **เล่นกับ async ในที่ปลอดภัยก่อน (ยังไม่แตะ TaskFlow):** ให้ผู้เรียนเปิด console เบราว์เซอร์ (F12 → Console) แล้วพิมพ์ทีละบรรทัดเพื่อ "เห็นการรอด้วยตา":
   ```js
   console.log("1 ก่อนสั่ง");
   fetch("https://zenquotes.io/api/random")
     .then(res => res.json())
     .then(data => console.log("3 ข้อมูลมาแล้ว:", data));
   console.log("2 หลังสั่ง (มาก่อนข้อมูล!)");
   ```
   ชี้ให้เห็นว่า log ขึ้นเรียง **1 → 2 → 3** ไม่ใช่ 1 → 3 → 2 — พิสูจน์ว่า `fetch` "ไม่หยุดรอ" บรรทัดถัดไปทำงานต่อทันที (นี่คือ asynchronous ของจริง). อธิบายว่า `.then()` คือ "พอเบอร์คิวเรียกค่อยทำอันนี้".

2. **เขียนใหม่ด้วย `async/await` ให้อ่านง่ายขึ้น:** บอกว่า `.then()` ซ้อน ๆ จะอ่านยาก เลยมีท่า `async/await` ที่อ่านเหมือนบรรทัดต่อบรรทัด. ให้ลองในไฟล์ทดลองหรือ console:
   ```js
   // ดึงคำคมจาก API แบบดิบ ๆ เพื่อลองเล่น async/await
   async function fetchQuote() {
     const res = await fetch("https://zenquotes.io/api/random"); // รอให้เซิร์ฟเวอร์ตอบ
     const data = await res.json();                              // รอแกะ JSON
     console.log("ได้:", data);
   }
   fetchQuote();
   ```
   ย้ำ: `await` 2 ครั้งเพราะมี "การรอ" 2 จังหวะ — (1) รอเซิร์ฟเวอร์ตอบ, (2) รอแกะ JSON.

3. **สร้าง Client Component `DailyQuote`:** สร้างไฟล์ `src/components/DailyQuote.jsx` (หรือ `.tsx` ถ้าโปรเจกต์เป็น TypeScript แล้วจาก Phase 4) ขึ้นต้นด้วย `"use client"` (เพราะต้องใช้ `useState` + `useEffect` ในเบราว์เซอร์). ตอนแรกให้ return โครงเปล่า ๆ แสดงคำว่า "การ์ดคำคม" ไปก่อน เพื่อยืนยันว่ามันโผล่บนหน้า Dashboard ได้.

4. **เตรียม 3 ก้อน state ด้วย `useState`:** อธิบายว่าเราต้อง "จำ" 3 อย่าง:
   ```js
   const [quote, setQuote] = useState(null);     // ข้อมูลคำคม (ตอนแรกยังไม่มี = null)
   const [loading, setLoading] = useState(true); // กำลังโหลดอยู่ไหม (เริ่ม true)
   const [error, setError] = useState(null);     // มี error ไหม (เริ่มไม่มี = null)
   ```
   เชื่อมโยงกับคอนเซปต์ 3 สถานะของ Promise (pending/fulfilled/rejected) ที่สอนไปแล้ว.

5. **ดึงข้อมูลใน `useEffect` พร้อมจัดการ loading/error เต็มรูปแบบ:** เขียนฟังก์ชัน async ข้างใน `useEffect` (เพราะ `useEffect` เองห้ามเป็น async ตรง ๆ — สอนเป็นกฎจำไปก่อน). ใส่ `try / catch / finally` ครบ:
   ```js
   useEffect(() => {
     // โหลดคำคมจาก API พร้อมจัดการ loading/error
     async function loadQuote() {
       try {
         setLoading(true);
         setError(null);
         const res = await fetch("https://zenquotes.io/api/random");
         if (!res.ok) throw new Error("เซิร์ฟเวอร์ตอบไม่ปกติ");
         const data = await res.json();
         setQuote({ text: data[0].q, author: data[0].a }); // ← หยิบ field ตามที่ดูไว้
       } catch (e) {
         setError("โหลดคำคมไม่สำเร็จ ลองใหม่อีกครั้งนะ");
       } finally {
         setLoading(false); // ไม่ว่าจะสำเร็จหรือพลาด ก็เลิกโหลดเสมอ
       }
     }
     loadQuote();
   }, []); // [] = ทำครั้งเดียวตอนหน้าเปิด
   ```
   อธิบายทีละบรรทัด โดยเฉพาะ:
   - **`res.ok`** = เช็คว่าเซิร์ฟเวอร์ตอบสำเร็จไหม (สำคัญ! `fetch` ไม่ throw error เองเวลาเซิร์ฟเวอร์ตอบ 404/500 เราต้องเช็คเอง).
   - **`throw new Error(...)`** ในกรณีไม่ ok = "โยน" ให้ `catch` รับ.
   - **`finally`** = "ไม่ว่ายังไงก็ทำ" — เหมาะกับปิด loading.
   - **`data[0].q`** ← ย้ำว่าตรงนี้ "ขึ้นกับ API ที่เลือก" ถ้าสลับ API ให้แก้ตรงนี้ (กลับไปดู JSON ที่ทดสอบไว้).

6. **วาด UI ตาม state (แสดง 3 สถานะให้เห็นจริง):** ใน return เลือกแสดงตามสถานะ — สอน conditional rendering (ทบทวนจาก React):
   ```jsx
   if (loading) return <p>กำลังโหลดคำคม...</p>;
   if (error) return (
     <div>
       <p>{error}</p>
       <button onClick={loadQuote}>ลองใหม่</button> {/* ฟังก์ชัน loadQuote เราจะสร้างเต็มในข้อ 9 */}
     </div>
   );
   return (
     <blockquote>
       “{quote.text}”
       <footer>— {quote.author}</footer>
     </blockquote>
   );
   ```
   จัดสไตล์ด้วย Tailwind ให้เป็นการ์ดสวย ๆ (ทบทวน `dark:` ให้เข้ากับ dark mode จาก Phase 6).

7. **วางการ์ดบนหน้า Dashboard:** import `DailyQuote` ไปใส่ใน `src/app/dashboard/page.*` (หน้าที่ทำไว้ตั้งแต่ Phase 6). เปิด `/dashboard` ในเบราว์เซอร์ — **โมเมนต์สำคัญ: เห็นคำคมสดจากอินเทอร์เน็ตขึ้นบนแอปของตัวเอง** และเห็นคำว่า "กำลังโหลด..." วาบก่อนข้อมูลมา.

8. **ทดสอบ "ตอนเน็ตพัง" ของจริง (อย่าข้าม — นี่คือหัวใจของ error state):** สอน 2 วิธีพิสูจน์ว่า error state ทำงาน:
   - **วิธีง่าย:** แก้ URL ใน `fetch` ให้พิมพ์ผิด (เช่นเติมตัวอักษรมั่ว) → save → ต้องเห็นข้อความ error + ปุ่มลองใหม่ ไม่ใช่จอขาว. แล้วแก้กลับให้ถูก.
   - **วิธีเหมือนจริง:** F12 → แท็บ Network → เลือก throttling เป็น "Offline" หรือ "Slow 3G" → รีเฟรช → เห็น loading ค้างนาน / เห็น error. อธิบายว่า "นี่คือสิ่งที่ผู้ใช้จริงเจอตอนเน็ตไม่ดี เราจึงต้องเผื่อไว้".

9. **เพิ่มปุ่ม "สุ่มคำคมใหม่" (เรียก async ซ้ำตามการกด):** ดึงตรรกะการโหลดออกมาเป็นฟังก์ชันที่เรียกซ้ำได้ (เช่น `loadQuote`) — ย้ายขึ้นมาประกาศ **ในตัว component** (ไม่ใช่ข้างใน `useEffect` แล้ว) เพื่อให้ทั้ง `useEffect` และปุ่มเรียกฟังก์ชันเดียวกันได้. รูปแบบสุดท้ายที่ต่อกันได้จริงหน้าตาแบบนี้:
   ```jsx
   // โหลดคำคมจาก API — ประกาศในตัว component เพื่อให้เรียกซ้ำได้
   const loadQuote = async () => {
     try {
       setLoading(true);
       setError(null);
       const res = await fetch("https://zenquotes.io/api/random");
       if (!res.ok) throw new Error("เซิร์ฟเวอร์ตอบไม่ปกติ");
       const data = await res.json();
       setQuote({ text: data[0].q, author: data[0].a });
     } catch (e) {
       setError("โหลดคำคมไม่สำเร็จ ลองใหม่อีกครั้งนะ");
     } finally {
       setLoading(false);
     }
   };

   useEffect(() => {
     loadQuote(); // โหลดครั้งแรกตอนหน้าเปิด
   }, []); // [] ว่าง = ทำครั้งเดียวตอน component mount ไม่ใช่ทุกครั้งที่ re-render

   // ...ใน return ปุ่มผูกกับฟังก์ชันเดียวกัน:
   // <button onClick={loadQuote}>สุ่มคำคมใหม่</button>
   ```
   กดปุ่ม → เห็น loading → ได้คำคมใหม่. (ปุ่ม "ลองใหม่" ในกรณี error ก็เรียก `loadQuote` ตัวเดียวกันนี้.) ตอกย้ำว่า "async ไม่ได้ทำครั้งเดียว เรียกกี่ครั้งก็ได้ตามต้องการ".

10. **(ทางเลือก/ต่อยอด) การ์ด "วันหยุดถัดไป" แนะนำ due date:** ถ้าผู้เรียนยังไหว ให้ทำซ้ำคอนเซปต์เดิมกับ public holiday API ฟรี (เช่น `https://date.nager.at/api/v3/PublicHolidays/2026/TH` — **ทดสอบ endpoint ในเบราว์เซอร์ก่อนเสมอ ตามขั้นตอนด้านบน**). เป้าหมาย: แสดงวันหยุดถัดไปบน Dashboard เพื่อ "แนะนำว่าอย่าตั้ง due date ตรงวันหยุด" — ไม่ต้องเชื่อมกับฟอร์มจริงก็ได้ แค่แสดงผลก็พอ. ย้ำ: เปลี่ยน API = เปลี่ยนแค่ URL + field ที่หยิบ.

- **อธิบายเป็นคำพูดได้** ว่า "การดึงข้อมูลจาก external API ต่างจาก Server Actions ยังไง" อย่างน้อย 2 ข้อ (เช่น คุยกับเซิร์ฟเวอร์ของคนอื่น, ต้องเผื่อ error/ช้าเสมอ).

### 📚 คอนเซปต์ที่จะสอน (พร้อมคำอธิบายสั้นๆ)
- **dev vs production:** ความต่างของ environment ที่ใช้พัฒนา กับที่ผู้ใช้จริงเข้าถึง; ทำไมต้องเปลี่ยน DB.

- **CI/CD เบื้องต้น (conceptual):** ทุกครั้งที่ push GitHub แล้ว Vercel deploy ให้เอง คือ pipeline แบบง่าย.
- **README & การเล่าโปรเจกต์:** README ที่ดีอธิบาย "ทำอะไร/เทคโนโลยีอะไร/รันยังไง/มีฟีเจอร์อะไร"; การเล่าตอนสัมภาษณ์เน้น "ปัญหาที่แก้ + การตัดสินใจ + สิ่งที่เรียนรู้".

### 👣 ลำดับขั้นจับมือทำ
1. **เตรียม production DB:** สมัคร Neon/Supabase, สร้าง database, คัดลอก connection string (ดู `SETUP.md` **§15**).
2. **ปรับ Prisma เป็น Postgres:** เปลี่ยน datasource, ตั้ง env สำหรับ prod, รัน migration บน DB ใหม่. ทดสอบ local ชี้ไป Postgres ว่ายังทำงาน.
3. **เตรียม repo ให้พร้อม deploy:** ตรวจว่า `.env` ไม่หลุดขึ้น git, ตรวจว่า build ผ่านในเครื่อง.
   > 💾 **ย้ำเรื่อง backup (ทบทวนจาก Phase 4):** `.env` กับ `dev.db` ไม่เคยขึ้น git มาตั้งแต่ต้น — `git push` จึง **ไม่ได้ backup ค่าความลับใน `.env` ให้**. ก่อน deploy ให้ผู้เรียน **จดค่าทั้งหมดใน `.env` เก็บไว้ที่ปลอดภัย** (เช่น password manager) เพราะเดี๋ยวต้องเอาไปกรอกซ้ำบน Vercel ในข้อ 4 และถ้าเครื่องพังจะได้ไม่หาย. ส่วนข้อมูลของจริงจะย้ายไปอยู่บน Postgres บน cloud (ข้อ 1–2) ซึ่งผู้ให้บริการ (Neon/Supabase) ดูแล backup ระดับหนึ่งให้ — ต่างจาก `dev.db` ที่เป็นแค่ข้อมูลทดสอบ local.
4. **deploy บน Vercel:** เชื่อม GitHub repo, **ตั้ง environment variables บน Vercel ให้ครบทุกค่าจาก `.env`** (เช่น `DATABASE_URL` ที่ชี้ไป Postgres production, `AUTH_SECRET` และ env อื่นที่ใช้) — **ห้ามใส่ความลับลงในโค้ดเด็ดขาด เพราะโค้ดขึ้น git แต่ env ไม่ขึ้น**. ถ้าลืมตั้งค่าใดค่าหนึ่ง deploy จะ build ผ่านแต่แอปพังตอนรัน (ดูวิธีตั้งใน `SETUP.md` §15). จากนั้นกด deploy — **โมเมนต์สำคัญ: ได้ URL จริง**. เปิดบนมือถือให้ผู้เรียนดู.
5. **ทดสอบ production:** สมัคร/ล็อกอิน/เพิ่ม task บนเว็บจริง ตรวจว่าทุกอย่างทำงาน; ไล่แก้ปัญหา deploy ที่พบบ่อย (env, migration).
6. **เขียน README ระดับมืออาชีพ:** ใส่ชื่อ, คำอธิบาย, screenshot/gif, tech stack (อ้าง stack ที่ใช้), ฟีเจอร์, วิธีรัน local, ลิงก์ live demo.
7. **ตั้งค่า custom touches (ถ้าอยาก):** favicon, ชื่อแอป, meta — ให้ดู finished.
8. **เตรียมเล่าโปรเจกต์ + next steps:** ร่างคำตอบสัมภาษณ์ ("ทำไมเลือก stack นี้", "เจอปัญหาอะไรแก้ยังไง", "ถ้ามีเวลาเพิ่มจะทำอะไรต่อ") และลิสต์ next steps ในเอกสาร.
